## Procedure INF-20260705-00: Pulumi state on Garage, and importing existing resources

This procedure documents how the Pulumi stacks in this repository store and access their state, what that implies for
disaster recovery, and how to bring hand-created ("clickops") resources under Pulumi **without recreating them** via
`pulumi import`.

It applies to every Pulumi project using the self-hosted backend — today `projects/chezmoi.sh/src/infrastructure/pulumi`
(`chezmoi-sh-infra`) and `projects/amiya.akn/src/infrastructure/pulumi` — whose `Pulumi.yaml` files all point at the
same backend:

```yaml
backend:
  url: "s3://pulumi-states?endpoint=s3.chezmoi.sh&region=fr-par-1"
```

### State backend layout

- **Backend**: the `pulumi-states` bucket on [Garage](https://garagehq.deuxfleurs.fr/), the S3-compatible object store
  running on `nas.chezmoi.sh` (see
  [`projects/chezmoi.sh/docs/TRUENAS.md`](../../../projects/chezmoi.sh/docs/TRUENAS.md)). There is no Pulumi Cloud
  involvement — state, history, and secrets all live in this bucket.
- **Secrets provider**: the default passphrase provider. Each project keeps its passphrase in a SOPS-encrypted
  `.mise/env.yaml` (`PULUMI_CONFIG_PASSPHRASE`, plus `PULUMI_STACK` for stack selection), decrypted transparently by
  `mise` when running from the project directory — the `pulumi:*` tasks never prompt.
- **S3 credentials**: the Pulumi CLI authenticates against Garage with the standard AWS environment variables
  (`AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`), holding a Garage **application key** allowed on the `pulumi-states`
  bucket. This credential is ambient operator configuration, not committed anywhere in the repo.

> \[!IMPORTANT] **The state's availability chain is longer than it looks.** Garage runs on the NAS
> (`zp1hs01/applications/truenas/fr.deuxfleurs.garage`), the NAS is a VM on `pve-01`, and `pve-01` is also what the
> stacks themselves manage. Losing the NAS (or the host under it) makes every `pulumi` command fail at `login` — plan
> around it:
>
> - **NAS down, host fine** → restore the NAS VM from its PBS backup first; Pulumi state comes back with it.
> - **Host lost** → follow [`PROXMOX-VE.md`](../../../projects/chezmoi.sh/docs/PROXMOX-VE.md), "Rebuild the host": the
>   NAS VM (and Garage with it) must be restored **before** any `pulumi up` can run.
> - **NAS storage lost too** → the Garage dataset is part of the daily off-site sync of
>   `/mnt/zp1hs01/applications/truenas` to Backblaze B2 (see `TRUENAS.md`, "Backups", Layer 3), so the state survives
>   site loss with at most one day of drift. Restore the dataset from B2, bring Garage up, then `pulumi login`.

### Prerequisites

- `mise install` ran in the target project directory (Node, `pulumi` CLI, SOPS key available via `SOPS_AGE_KEY_FILE`).
- A Garage application key for `pulumi-states` exported as `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`.
- For `projects/chezmoi.sh` specifically: some resources need provider credentials beyond the defaults — notably every
  Proxmox VE **ACME** resource requires a real `root@pam` username/password session (API tokens are hard-rejected by
  those endpoints); see
  [`stack/proxmox/README.md`](../../../projects/chezmoi.sh/src/infrastructure/pulumi/stack/proxmox/README.md),
  "Bootstrapping", for the Keychain-based pattern.

### Minting (or rotating) the state credential

If the application key is lost or needs rotation, mint a new one from the Garage CLI on the NAS (inside the Garage app's
container):

```sh
garage key create pulumi-state-<year>
garage bucket allow pulumi-states --read --write --key pulumi-state-<year>
garage key deny ... / garage key delete ...   # then retire the old key
```

Export the new key pair as `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` and verify with `pulumi stack ls` from any
project directory.

### Day-to-day state operations

All commands run from the project directory (so `mise` injects `PULUMI_STACK` / `PULUMI_CONFIG_PASSPHRASE`):

```sh
pulumi stack                    # current stack + resource summary
pulumi stack export > state.json   # full state snapshot — the fastest way to inspect URNs, IDs, outputs
pulumi state taint <urn>        # mark a resource for replacement on next apply
pulumi state delete <urn>       # drop a resource from state without touching the real object
pulumi refresh                  # reconcile state with live reality (the pulumi:* tasks already pass --refresh)
```

`pulumi stack export` is also the way to find the exact URN the `state`/`import` commands expect — never guess one.

### Importing an existing resource (zero recreation)

Use this whenever an object already exists on the target system (created by hand, or by a recipe predating the stack)
and should come under Pulumi management without a destroy/create cycle.

1. **Declare the resource in code first**, matching the live object's settings as closely as possible. Keep it flat —
   **do not set the `parent` resource option**: imports done without a matching `--parent` produce a different URN than
   a `parent`-declared resource, which later shows up as a spurious delete+create (this bit the Proxmox VE stack; see
   `stack/proxmox/README.md`, "Adding a new identity or ACL binding").

2. **Import it into state**, using the provider's documented ID format for that type:

   ```sh
   pulumi import <type-token> <name> <id> --yes
   # e.g. pulumi import proxmox:index/virtualEnvironmentRole:VirtualEnvironmentRole exporter Exporter
   ```

   The type token and ID format are in each provider's resource documentation (for bridged providers, the upstream
   Terraform provider's import docs apply).

3. **Converge to a zero diff.** Run `pulumi preview` and adjust the code until the imported resource shows **no
   changes**. Never accept a preview that replaces the resource you just imported — that is the recreation this
   procedure exists to avoid.

4. **Drop the automatic protection.** `pulumi import` marks resources `protect: true` in state. Once the diff is clean,
   align it with the rest of the stack (this repo does not use `protect` by default):

   ```sh
   pulumi state unprotect <urn>
   ```

5. **Apply** (`mise run pulumi:diff` / `mise run pulumi:apply` — with whatever provider credentials the resource needs,
   e.g. the `root@pam` env pair for Proxmox VE ACME resources).
