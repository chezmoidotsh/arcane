## Procedure: Upgrading a Proxmox NixOS LXC (rootfs swap)

This procedure upgrades a Proxmox LXC container to a new image by spinning up a parallel CT with the new template, then atomically moving the persistent data volume(s) (`mp0`, `mp1`, …) from the old CT to the new one. The data volumes are never copied — only their attachments are reassigned, so there is **zero data risk**.

Two CTs co-exist briefly but only one ever owns each mount point at a time, so services sharing the store cannot corrupt it.

**Target downtime:** ≤ 2 minutes (stop old → `pct set` swap → start new → services come up). Validated at \~19 s on the live o11y appliance.

### Technical framework

* Storage backend: LVM-thin (`nvme-lvm`). Mount points use volumes like `nvme-lvm:vm-<VMID>-disk-N`, mounted inside the container. Uid mapping: unprivileged containers map container uid N → host uid `100000 + N`.
* Template baked by `mise run lxc:build` and pushed by `mise run lxc:push -- <pve_host>`.
* All commands run on the Proxmox host as `root`. Connect via SSH from your workstation.

### Prerequisites

* The new template is already on the PVE host under `/var/lib/vz/template/cache/<LXC_NAME>.<VERSION>-amd64.tar.xz` (see Step 0 if not).
* The target VMID (`TARGET_ID`, default `199`) is free, or you're prepared to `pct destroy` a leftover from a previous run.
* Confirmed that the new image's service uid/gid are unchanged from the running CT. If they change, the chown step at the end of cutover becomes mandatory.

Export them once in your shell to keep the rest of the procedure copy-pastable:

```sh
export PVE_HOST=pve-01.pve.chezmoi.sh
export SOURCE_ID=101
export TARGET_ID=199
export VERSION=2026.06.05-1
export LXC_NAME=observability
export TEMPLATE=/var/lib/vz/template/cache/${LXC_NAME}.${VERSION}-amd64.tar.xz
```

***

## Step 0 — Build and push the new template (skip if already on PVE)

If the template is already in `/var/lib/vz/template/cache/`, jump to Step 1.

> **git-tracked vs git-staged content** — Nix flakes only see file content that is **staged** in the git index. Files added with `git add --intent-to-add` (the default after `--intent-to-add` or after creating an untracked file) appear as empty placeholders to Nix — the build will silently use the previous (or empty) content and produce a tarball that looks correct but has the OLD config baked in. Always `git add <files>` (no flag) before `mise run lxc:build`.

```sh
cd ${BUILD_PATH}

# Stage everything you intend to ship. Verify nothing important is still "AM"
# (added-with-intent + modified) or untracked.
git status --short config/ modules/ flake.nix
git add config/ modules/ flake.nix
git status --short config/ modules/ flake.nix   # should be "M " or "A " (staged)

# Bump the version (CalVer YYYY.MM.DD with optional -N suffix for same-day rebuilds,
# or SemVer depending on the LXC's versioning scheme)
vim flake.nix   # update `version = "…"` then save

mise run lxc:build
mise run lxc:push -- ${PVE_HOST}
```

***

## Step 1 — Pre-flight checks

Confirm template, source config, and target id. Nothing here changes any state.

```sh
# Template present on PVE
ssh root@${PVE_HOST} "test -f ${TEMPLATE} && ls -lh ${TEMPLATE}"

# Source config (capture rootfs + mp0 lines)
ssh root@${PVE_HOST} "cat /etc/pve/lxc/${SOURCE_ID}.conf"

# Target id free? If a leftover CT exists from a prior run, destroy it now —
# the procedure assumes a clean slate. pct destroy will refuse to touch CT
# ${SOURCE_ID}'s volumes (mp0 still owned), so it's safe.
ssh root@${PVE_HOST} "
if [ -f /etc/pve/lxc/${TARGET_ID}.conf ]; then
  echo 'WARN: CT ${TARGET_ID} exists — destroying.'
  pct stop ${TARGET_ID} 2>/dev/null || true
  pct destroy ${TARGET_ID}
fi
test ! -f /etc/pve/lxc/${TARGET_ID}.conf && echo TARGET FREE"
```

Stash the mp0 spec for reuse in Step 5:

```sh
MP0_SPEC=$(ssh root@${PVE_HOST} "grep '^mp0:' /etc/pve/lxc/${SOURCE_ID}.conf | sed 's/^mp0: //'")
echo "MP0_SPEC=${MP0_SPEC}"
# Expected example: nvme-lvm:vm-101-disk-1,mp=/var/lib/o11y,size=64G
```

***

## Step 2 — Create the target CT (rootfs only, no mp0)

Create the new CT with a fresh rootfs from the template. **Do not attach mp0 yet** — it still belongs to the source. Mirror the source's `features`, `memory`, `cores` so the smoke test in Step 3 reflects production conditions.

> **`features: nesting=1` is mandatory.** NixOS LXCs need it for stage-2 activation. Without it, the CT boots a barebones `/bin/sh`, `/run/current-system` is never created, and the smoke test will report "system is degraded" with no usable PATH. `pct create`'s default has *no* features.

```sh
ssh root@${PVE_HOST} "
ROOTFS_LINE=\$(grep '^rootfs:' /etc/pve/lxc/${SOURCE_ID}.conf | sed 's/^rootfs: //')
ROOTFS_STORAGE=\$(echo \"\$ROOTFS_LINE\" | cut -d: -f1)
ROOTFS_SIZE=\$(echo \"\$ROOTFS_LINE\" | grep -oE 'size=[0-9]+' | cut -d= -f2)
FEATURES=\$(grep '^features:' /etc/pve/lxc/${SOURCE_ID}.conf | sed 's/^features: //')
MEMORY=\$(grep '^memory:' /etc/pve/lxc/${SOURCE_ID}.conf | awk '{print \$2}')
CORES=\$(grep '^cores:' /etc/pve/lxc/${SOURCE_ID}.conf | awk '{print \$2}')
echo \"create: rootfs=\$ROOTFS_STORAGE:\$ROOTFS_SIZE features=\$FEATURES memory=\$MEMORY cores=\$CORES\"

pct create ${TARGET_ID} ${TEMPLATE} \
  --rootfs \$ROOTFS_STORAGE:\$ROOTFS_SIZE \
  --features \"\$FEATURES\" \
  --memory \$MEMORY \
  --cores \$CORES \
  --swap 0
"
```

The new CT now has its own `nvme-lvm:vm-${TARGET_ID}-disk-0` rootfs, the same activation-critical features as the source, and a default network (auto-generated MAC) suitable for the Step 3 smoke test.

***

## Step 3 — Smoke-test the new template (default network, no data)

The freshly-created CT has its own auto-generated MAC and no mp0. Boot it as-is to confirm the new template's OS and basic networking work — **before** we copy the source config (which would clone the source's MAC and cause a bridge collision while the source is still running).

```sh
ssh root@${PVE_HOST} "pct start ${TARGET_ID}"
sleep 10

# Sanity: /run/current-system must exist. If missing, stage-2 activation
# failed (almost always: features=nesting=1 was not set in Step 2).
ssh root@${PVE_HOST} "pct exec ${TARGET_ID} -- /bin/sh -c '
test -e /run/current-system && echo \"OK activation\" || echo \"FAIL — no /run/current-system\"
'"

# Now use the full NixOS PATH for the rest of the checks.
ssh root@${PVE_HOST} "pct exec ${TARGET_ID} -- /run/current-system/sw/bin/bash -lc '
echo === is-system-running ===
systemctl is-system-running
echo === failed units ===
systemctl --failed --no-legend
'"
```

Expected:

* `OK activation` from the first probe.
* `systemctl is-system-running` returns `degraded`. **This is fine** — the LXC always has a baseline of expected failures:
  * `sys-kernel-debug.mount` — unprivileged containers can't mount `/sys/kernel/debug`
  * `zfs-import-scan.service` — no ZFS in the LXC
  * `systemd-journal-upload.service` — package not installed
* Application services are *inactive* rather than failed: they tried to start, exited because `${MP0_MOUNT}` is empty (no mp0), and got hit by systemd's start-rate-limit. This is expected at this stage.

Stop the CT before merging configs:

```sh
ssh root@${PVE_HOST} "pct stop ${TARGET_ID}"
```

***

## Step 4 — Merge source config into target (rootfs swap, drop mp0)

Now copy every line of the source config into the target, replacing only the source rootfs volume name with the freshly-allocated one, and strip `mp0` so the target can never accidentally dual-mount the volume that still belongs to the source. After this step the target shares the source's MAC, hostname, VLAN, etc. — it must **not be started** until Step 5's cutover.

```sh
ssh root@${PVE_HOST} "
NEW_ROOTFS_VOL=\$(grep '^rootfs:' /etc/pve/lxc/${TARGET_ID}.conf | awk '{print \$2}' | cut -d, -f1)
OLD_ROOTFS_VOL=\$(grep '^rootfs:' /etc/pve/lxc/${SOURCE_ID}.conf | sed 's/^rootfs: //' | cut -d, -f1)
echo \"OLD=\$OLD_ROOTFS_VOL NEW=\$NEW_ROOTFS_VOL\"
sed \"s|\$OLD_ROOTFS_VOL|\$NEW_ROOTFS_VOL|g\" /etc/pve/lxc/${SOURCE_ID}.conf > /etc/pve/lxc/${TARGET_ID}.conf
sed -i '/^mp0:/d' /etc/pve/lxc/${TARGET_ID}.conf
[ -f /etc/pve/firewall/${SOURCE_ID}.fw ] && cp /etc/pve/firewall/${SOURCE_ID}.fw /etc/pve/firewall/${TARGET_ID}.fw || echo '(no firewall file to copy)'
echo === target conf ===
cat /etc/pve/lxc/${TARGET_ID}.conf
"
```

Expected: target config matches source line-for-line minus the `mp0` line, with the new rootfs volume name.

***

## Step 5 — Cutover (downtime starts here)

Stop the source, detach `mp0` from it, attach the same volume to the target, then start the target. `pct set --delete mp0` only removes the *reference* from the config — the underlying LVM volume is preserved (it moves to an `unused0:` entry in the source config, which we clean up in Step 7).

```sh
T0=$(date +%s)
ssh root@${PVE_HOST} "
pct stop ${SOURCE_ID}
pct set ${SOURCE_ID} --delete mp0
pct set ${TARGET_ID} -mp0 '${MP0_SPEC}'
pct start ${TARGET_ID}
"
echo "cutover commands: $(( $(date +%s) - T0 ))s — services still warming up"
```

Verify data-volume ownership (only needed if the CT uid changed between images — usually not):

```sh
# Confirm host-side ownership still matches container uid → host mapping
ssh root@${PVE_HOST} "pct mount ${TARGET_ID} >/dev/null && stat -c '%u:%g %n' /var/lib/lxc/${TARGET_ID}/rootfs${MP0_MOUNT}; pct unmount ${TARGET_ID}"
```

If ownership is wrong, fix it once (CT must be stopped):

```sh
ssh root@${PVE_HOST} "pct stop ${TARGET_ID} && pct mount ${TARGET_ID} && chown -R ${HOST_UID}:${HOST_UID} /var/lib/lxc/${TARGET_ID}/rootfs${MP0_MOUNT} && pct unmount ${TARGET_ID} && pct start ${TARGET_ID}"
```

***

## Step 6 — Verify the new CT is healthy (downtime ends when this passes)

Give services \~15 s to come up after `pct start`, then run the health sweep.

### Generic health check

```sh
sleep 15
ssh root@${PVE_HOST} "pct exec ${TARGET_ID} -- /run/current-system/sw/bin/bash -lc '
echo === is-system-running ===
systemctl is-system-running
echo === failed units ===
systemctl --failed --no-legend
'"
```

Expected:

* `degraded` overall — only `sys-kernel-debug.mount`, `zfs-import-scan.service`, `systemd-journal-upload.service` failed (same baseline as Step 3).
* No new failures compared to Step 3.

***

## Step 7 — Decommission the old CT

If verification passes, destroy the source. `pct set --delete mp0` from Step 5 moved the volume reference to `unused0:` in the source config — PVE still considers `vm-${SOURCE_ID}-disk-1` "owned" by the source VMID by naming convention, so `pct destroy` will refuse to complete while that entry exists (`lvremove` fails: "Logical volume … contains a filesystem in use"). Same for `pct set --delete unused0`.

Strip the dangling references via direct config edit, then destroy:

```sh
ssh root@${PVE_HOST} "
sed -i '/^rootfs:/d; /^unused0:/d' /etc/pve/lxc/${SOURCE_ID}.conf
pct destroy ${SOURCE_ID}
pct list | head -10
"
```

The persistent volume's LVM name keeps the old VMID prefix (`vm-${SOURCE_ID}-disk-1`) — that's cosmetic. Renaming would require stopping the target, `lvrename`, `pct set` to point at the new name, then restarting. Not worth the extra 30–60 s of downtime; the name just records which VMID first allocated the volume.

Clean up any orphan LVs from previous failed operations:

```sh
ssh root@${PVE_HOST} "lvs nvme-lvm | grep vm-${SOURCE_ID}-"
# Expect: only vm-${SOURCE_ID}-disk-1 (the live mp0, "Vwi-aotz--" — note the 'o' = open).
# If you see other vm-${SOURCE_ID}-disk-N volumes, confirm they're not in any
# /etc/pve/{lxc,qemu-server}/*.conf, then:
#   ssh root@${PVE_HOST} "lvremove -f nvme-lvm/vm-${SOURCE_ID}-disk-0"
```

Optionally rename the target VMID back to the canonical id. PVE does not support in-place renames, so the swap is a full re-run of this procedure with a third CT. Easiest path: leave the new CT under `TARGET_ID` and update any consumer that hardcoded the original ID.

***

## Rollback

If Step 6 fails, reverse the move. Same downtime budget (\~20 s).

```sh
ssh root@${PVE_HOST} "
pct stop ${TARGET_ID}
pct set ${TARGET_ID} --delete mp0
pct set ${SOURCE_ID} -mp0 '${MP0_SPEC}'
pct start ${SOURCE_ID}
"

# Confirm the old CT is back
sleep 15
ssh root@${PVE_HOST} "pct exec ${SOURCE_ID} -- /run/current-system/sw/bin/systemctl is-system-running"

# Destroy the failed target — same gotcha as Step 7: pct set --delete mp0
# leaves an unused0 entry that refers to the now-attached-elsewhere volume.
ssh root@${PVE_HOST} "
sed -i '/^unused0:/d' /etc/pve/lxc/${TARGET_ID}.conf
pct destroy ${TARGET_ID}
"
```

The persistent data volume never left its storage — it was only re-referenced by the configs.

## Recap of gotchas

* **`git add --intent-to-add` is invisible to Nix flake builds.** Always fully `git add` source-tree changes before `mise run lxc:build`, or the tarball ships the previous content even though the build "succeeds".
* **`features: nesting=1` is mandatory for NixOS LXCs.** `pct create` doesn't inherit it from the source — set it explicitly in Step 2.
* **`pct create` defaults to 512 MiB / 1 core.** Mirror the source's `memory` and `cores` (Step 2) so the smoke test reflects production resource pressure.
* **The MAC is shared between source and target once Step 4 runs.** Never `pct start` the target after Step 4 except as part of Step 5's cutover — the bridge would see a MAC collision and DHCP would offer the same IP to both.
* **`pct destroy` tries to remove every `vm-<VMID>-*` volume**, even ones moved to `unused0:` that are actively mounted by another CT. The `lvremove` fails safely, but leaves the source CT in a half-destroyed state. Strip `rootfs:` and `unused0:` from the config first (Step 7).
* **The `mp0` LVM volume keeps the original `vm-<SOURCE_ID>-disk-1` name forever** after the swap — Proxmox doesn't rename LVs automatically. Cosmetic only; readable via the volume's "open" flag in `lvs` output.
