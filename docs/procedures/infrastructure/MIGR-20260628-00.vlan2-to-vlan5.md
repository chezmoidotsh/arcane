## Procedure MIGR-20260628-00: Migrate Talos nodes from VLAN 2 to VLAN 5

This procedure migrates existing Omni-managed Talos VMs from the legacy VLAN 2
(`eth0` tag) to VLAN 5 (the Homelab VLAN, standardized in ADR-014 V2) and adds
the second NIC (`eth1`) on the shared `talosnet` SDN VNet.

The migration is required because:

* Machine classes in `projects/chezmoi.sh/src/infrastructure/omni/machineclasses/`
  were updated from V1 (VLAN 2, single NIC) to V2 (VLAN 5, dual NIC).
* Omni **does not re-provision running VMs** when a machine class is updated —
  the class change only affects newly provisioned nodes.
* Existing nodes remain on VLAN 2 with a single NIC until manually migrated.

Running a mixed cluster (some nodes on VLAN 2, others on VLAN 5) is not
supported and causes split-network failures once `omnictl apply` pushes the new
class definition.

### Prerequisites

* [INF-20260627-00](INF-20260627-00.proxmox-sdn-setup.md) completed: the
  `talosnet` VNet and VLAN 5 trunk are live.
* PVE trunk on `vmbr1` carries **both VLAN 2 and VLAN 5** during the migration
  window. Do not remove VLAN 2 from the trunk until all nodes are migrated.
* `omnictl` authenticated (`mise run bao:login:admin` from `projects/chezmoi.sh/`).
* Cluster workloads tolerate a rolling node drain (one worker at a time).

### Required inputs

| Input       | Value                                  | Source                    |
| ----------- | -------------------------------------- | ------------------------- |
| Cluster     | `lungmen-akn` (or target cluster name) | Omni                      |
| Old VLAN    | `2`                                    | V1 machine class          |
| New VLAN    | `5`                                    | V2 machine class, ADR-014 |
| eth1 bridge | `talosnet`                             | INF-20260627-00           |

***

## Step 1 — Apply updated machine classes

Push the V2 machine class definitions to Omni. This updates Omni's class catalog
but does **not** affect running nodes.

```sh
# From projects/chezmoi.sh/
mise run omni:machineclass:diff    # preview — verify VLAN 5 and additional_nics appear
mise run omni:machineclass:apply   # push V2 definitions
```

Verify the updated class is reflected in Omni:

```sh
omnictl get machineclasses c1.small -o yaml | grep -A5 'providerdata'
# → vlan: 5, additional_nics: [{bridge: talosnet, ...}]
```

***

## Step 2 — Identify nodes to migrate

List all nodes currently registered in the cluster and confirm their VLAN:

```sh
omnictl get clustermachines -l omni.sidero.dev/cluster=lungmen-akn
```

For each node, confirm the current network config in Proxmox (PVE UI or SSH):

```sh
ssh root@pve-01.pve.chezmoi.sh
# For each VM ID (e.g. 200):
qm config 200 | grep -E 'net|vlan'
# V1 shows:  net0: virtio=...,bridge=vmbr1,tag=2
# V2 expects: net0: virtio=...,bridge=vmbr1,tag=5,firewall=0
#             net1: virtio=...,bridge=talosnet,firewall=0
```

***

## Step 3 — Migrate nodes one at a time (rolling)

Migrate one worker node at a time to preserve cluster availability. Drain the
node first to evict workloads gracefully.

### 3a — Drain the node

```sh
# Identify the node's Kubernetes name
kubectl get nodes

# Drain (evict pods, mark unschedulable)
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data
```

### 3b — Wipe and re-provision via Omni

The cleanest migration path is to wipe and re-provision the node through Omni.
Omni will provision a new VM using the updated V2 machine class (VLAN 5 + eth1).

In the Omni UI or via `omnictl`:

1. Remove the node from the cluster machine set (this triggers VM deletion by
   the Proxmox provider).
2. Wait for the VM to be deleted in PVE.
3. Re-add the node slot to the machine set — Omni provisions a fresh VM using
   the current machine class (now V2).

Alternatively, if the VM must be preserved (persistent local state), update the
network config in PVE directly then reboot:

```sh
ssh root@pve-01.pve.chezmoi.sh
# Change VLAN tag and disable firewall on net0:
qm set <vmid> --net0 virtio=<mac>,bridge=vmbr1,tag=5,firewall=0
# Add eth1 on talosnet:
qm set <vmid> --net1 virtio,bridge=talosnet,firewall=0
# Reboot the VM:
qm reboot <vmid>
```

> \[!WARNING]
> Manual `qm set` bypasses Omni's state — Omni will reconcile and may overwrite
> the NIC config on the next machine class sync. Prefer the wipe-and-reprovision
> path when possible.

### 3c — Verify node rejoins cluster

After re-provisioning, confirm the node is Ready and on VLAN 5:

```sh
kubectl get nodes <node-name>
# → Ready

# Verify eth0 VLAN tag and eth1 presence in the Talos node config:
talosctl -n <node-ip> get addresses
# → eth0: no IP (VLAN 5 trunk, Cilium L2 only)
# → eth1: 10.128.0.x (DHCP from talosnet)
```

### 3d — Un-cordon the node

```sh
kubectl uncordon <node-name>
```

Repeat Steps 3a–3d for each remaining worker node, then for the control plane
(if the cluster has more than one CP node). For a single-CP cluster,
`allowSchedulingOnControlPlanes: false` is set — the CP taint stays active and workloads
become unschedulable for the duration of the wipe. Migrate the CP last (Step 4) and
schedule it during a maintenance window.

***

## Step 4 — Migrate the control plane node

Follow the same drain → wipe/reprovision → verify sequence as Step 3. For a
single-node control plane, downtime is unavoidable during the wipe; schedule
during a maintenance window.

After the control-plane rejoins, verify the Kubernetes API is reachable and etcd
is healthy:

```sh
kubectl get nodes
talosctl -n <cp-ip> etcd members  # componentstatuses is removed in k8s 1.19+
```

***

## Step 5 — Remove VLAN 2 from the PVE trunk (optional, post-migration)

Once all nodes are on VLAN 5 and the cluster is fully healthy, VLAN 2 can be
removed from the `vmbr1` trunk if no other devices require it. This is optional
and can be done at any time after migration is complete.

```sh
# Verify no VMs still use VLAN 2:
ssh root@pve-01.pve.chezmoi.sh
for id in $(qm list | awk 'NR>1 {print $1}'); do
  echo "VM $id:"; qm config $id | grep 'tag=2'
done
```

If the output is empty, VLAN 2 is safe to remove from the trunk.

***

## Quick verifications

* All nodes `Ready`: `kubectl get nodes` → all `Ready`.
* eth1 DHCP on every node: `talosctl -n <ip> get addresses` → `eth1` has `10.128.0.x`.
* No nodes on VLAN 2: `qm list` + `qm config <id>` → no `tag=2` on any VM.
* Cilium healthy: `cilium status` → all components green.
* CSI working: create a test PVC and verify it mounts.
