## Procedure INF-20260525-01: Upgrading Kubernetes on a Talos Single-Node Cluster

This procedure details how to upgrade the Kubernetes version on a Talos-managed single-node cluster. It uses the `talosctl upgrade-k8s` command, which is the recommended and only supported method for Kubernetes upgrades on Talos — manual `kubeadm` or `kubectl` component patches are not applicable.

This procedure applies to a **single-node cluster** (combined control plane and worker). The `upgrade-k8s` command is safe to re-run if interrupted and streams its own progress, making it straightforward to recover from partial upgrades.

### Technical framework and conventions

All commands are executed through the `mise` environment, which sets `TALOSCONFIG` and `KUBECONFIG` automatically. The `talosctl` binary used here is the repository wrapper located in `scripts/talosctl`.

The `talosctl upgrade-k8s` command executes the Kubernetes upgrade in six sequential phases:

| Phase | Name                              | Description                                                                                                |
| ----- | --------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| 1     | Image pre-pulling                 | Downloads new component images to the node before any change is applied                                    |
| 2     | Control plane configuration       | Patches machine config; `kubelet` detects new static pod definitions and restarts control plane components |
| 3     | Proxy update                      | Updates the `kube-proxy` DaemonSet image version                                                           |
| 4     | Kubelet upgrade                   | Updates kubelet on all nodes; verifies restart via the `Node` resource                                     |
| 5     | Bootstrap manifest re-application | Applies updated bootstrap manifests                                                                        |
| 6     | Resource pruning                  | Removes resources that are no longer needed (Talos v1.13+; use `--manifests-no-prune` to skip)             |

On a single-node cluster, `--context` always targets the single node, which acts as both control plane and worker. Kubelet upgrade in phase 4 may briefly restart workloads; this is non-disruptive from the Talos perspective but causes a short pod restart cycle.

### Prerequisites

Before starting, ensure the following tools are installed and accessible via the `mise` environment: `talosctl`, `kubectl`.

You must also have:

* Network access to the node (IP reachable from your workstation).
* `TALOSCONFIG` and `KUBECONFIG` correctly set — verify with `talosctl version` and `kubectl get nodes`.
* Confirmed that the target Kubernetes version is supported by the **currently running Talos version**. Consult the [Talos support matrix](https://www.talos.dev/v1.13/introduction/support-matrix/) before proceeding. Upgrading to an unsupported Kubernetes version will fail.
* Talos OS is already at the version required to support the target Kubernetes version. If a Talos upgrade is also needed, complete [INF-20260525-00](./INF-20260525-00.upgrade-talos.md) first.

> \[!IMPORTANT]
> **Upgrade path for Kubernetes**
>
> Kubernetes supports upgrading one minor version at a time (e.g., `1.34.x` → `1.35.x`). Do not attempt to skip minor versions. If you need to go from `1.34.x` to `1.36.x`, run this procedure twice: once targeting `1.35.x`, then once targeting `1.36.x`.

### Required inputs

* `CLUSTER`: The talosconfig context name for the cluster (e.g., `amiya.akn`).
* `TARGET_K8S_VERSION`: Target Kubernetes version in the form `1.XX.Y` (e.g., `1.35.0`).

***

## Step 1 — Verify current state and check compatibility

Confirm the running Kubernetes version and validate that the target version is compatible with the current Talos version.

```sh
# Check running Kubernetes and Talos versions
kubectl version
talosctl version --context <CLUSTER>

# Check current node state
kubectl get nodes -o wide
```

If you need the node IP for any reason, retrieve it from the context:

```sh
# List all contexts with their configured endpoints and nodes
talosctl config contexts

# Or list cluster members (shows IP, hostname, role)
talosctl --context <CLUSTER> get members
```

> \[!NOTE]
> `talosctl config info --context <CLUSTER>` does not work with the `scripts/talosctl` wrapper in this repo. Use `talosctl config contexts` or `get members` instead.

Cross-reference the output against the [Talos support matrix](https://www.talos.dev/v1.13/introduction/support-matrix/). Locate the row for your current Talos version and confirm that `<TARGET_K8S_VERSION>` appears in the supported Kubernetes column.

> \[!IMPORTANT]
> **Do not proceed** if the target Kubernetes version is not listed as supported for your current Talos version. Upgrade Talos first (see [INF-20260525-00](./INF-20260525-00.upgrade-talos.md)), then return to this procedure.

> \[!IMPORTANT]
> **Check user-supplied feature gates against the Kubernetes changelog**
>
> `talosctl upgrade-k8s` only validates flags for the components it manages natively. It does
> **not** inspect `extraArgs`, `extraConfig`, or `featureGates` entries injected through the
> Talos machine config patch (e.g., `lungmen-akn-01.patch-config.yaml`).
>
> Before running the upgrade, open the patch config and manually verify every gate listed in:
>
> * `machine.kubelet.extraConfig.featureGates`
> * `cluster.apiServer.extraArgs.feature-gates`
>
> Cross-reference each gate against the [Kubernetes CHANGELOG](https://github.com/kubernetes/kubernetes/blob/master/CHANGELOG/)
> for `<TARGET_K8S_VERSION>`. A gate that has been **removed** or **graduated to GA** in the
> target version will cause an `unrecognized feature gate` panic at kube-apiserver and kubelet
> startup, taking the cluster offline. See post-mortem
> `docs/incidents/2026-05-26-lungmen-k8s-1.35-removed-featuregate.md`.

## Step 2 — Dry-run the upgrade

Run the upgrade in dry-run mode to preview all changes that will be applied without making any modifications to the cluster.

```sh
talosctl --context <CLUSTER> upgrade-k8s \
  --to <TARGET_K8S_VERSION> \
  --dry-run
```

Review the output carefully:

* Each phase is listed with the actions it will take.
* Image tags show which component images will be pulled.
* Machine config patches are described (not applied).

> \[!NOTE]
> The dry-run output is informational only — no images are pulled and no configuration is changed. Use it to estimate the scope of the upgrade and identify any unexpected changes before committing.

## Step 3 — Execute the upgrade

Run the upgrade. The command streams its own progress phase by phase and blocks until completion.

```sh
talosctl --context <CLUSTER> upgrade-k8s --to <TARGET_K8S_VERSION>
```

The command will print each phase as it starts and confirm completion. Example output structure:

```
patching node configs to Kubernetes 1.XX.Y...
phase 1: pre-pulling images ...
phase 2: updating control plane ...
phase 3: updating kube-proxy ...
phase 4: updating kubelet ...
phase 5: updating manifests ...
phase 6: pruning resources ...
upgrade to Kubernetes 1.XX.Y complete
```

> \[!IMPORTANT]
> **Do not interrupt the command** while it is running unless a phase is clearly hung. If the command is interrupted (e.g., terminal closed, network drop), it is safe to re-run the same command — it detects the current state and resumes from the appropriate phase.

> \[!NOTE]
> **Phase 4 workload restarts**
>
> When the kubelet is updated in phase 4, running pods may be briefly restarted. This is expected behavior. On a single-node cluster, this applies to both system pods and user workloads. The restart window is typically under one minute.

### Optional: skip resource pruning

If you want to preserve all existing resources and skip phase 6 (removal of deprecated resources), add the `--manifests-no-prune` flag:

```sh
talosctl --context <CLUSTER> upgrade-k8s \
  --to <TARGET_K8S_VERSION> \
  --manifests-no-prune
```

## Step 4 — Post-upgrade verification

After the command completes, verify that the node is reporting the new Kubernetes version and that all system pods are healthy.

```sh
# Verify node version
kubectl get nodes -o wide

# Verify all kube-system pods are healthy
kubectl get pods -n kube-system

# Verify Kubernetes API server version
kubectl version
```

Expected outcome:

* `kubectl get nodes` shows `Ready` with the new Kubernetes version in the `VERSION` column.
* All `kube-system` pods are `Running` or `Completed`, with no `CrashLoopBackOff` or `Error` states.
* `kubectl version` reports the new server version.

## Step 5 — Troubleshooting: interrupted or stalled upgrade

If `talosctl upgrade-k8s` was interrupted or is not making progress, first check the current state of cluster components:

```sh
# Check control plane component versions
kubectl get pods -n kube-system -o wide

# Check node kubelet version
kubectl get nodes -o custom-columns='NAME:.metadata.name,KUBELET:.status.nodeInfo.kubeletVersion'

# Inspect a specific stuck pod
kubectl describe pod <POD_NAME> -n kube-system

# Check Talos machine config for current static pod definitions
talosctl get staticpodstatus --context <CLUSTER>
```

If all components appear healthy but the upgrade command exited early, re-run the upgrade command with the same target version — it is idempotent and will resume from where it left off:

```sh
talosctl --context <CLUSTER> upgrade-k8s --to <TARGET_K8S_VERSION>
```

> \[!TIP]
> If a phase is genuinely stuck, check whether the issue is an image pull failure (network or registry access) or a pod crash. Image pull failures can be resolved by waiting for connectivity to restore and re-running the command. Pod crashes require inspecting logs with `kubectl logs <POD_NAME> -n kube-system`.

> \[!WARNING]
> **Remediating a broken machine config: `talosctl patch machineconfig` cannot delete keys**
>
> If a component crashes due to an unrecognized flag or feature gate, you may be tempted to
> remove it by editing the patch file and re-running `talosctl patch machineconfig`. This will
> **not work** — that command applies a strategic merge: it adds and updates keys but never
> deletes an existing key from the running node config.
>
> To delete a key from the running machine config:
>
> ```sh
> # 1. Extract the full running config
> talosctl get machineconfig --context <CLUSTER> -o yaml > /tmp/mc.yaml
>
> # 2. Edit /tmp/mc.yaml — remove the offending key entirely
>
> # 3. Re-apply as a full overwrite (not a merge)
> talosctl apply-config --context <CLUSTER> --file /tmp/mc.yaml
> ```
>
> `talosctl` stays operational even when kube-apiserver is down — `talosctl containers -k`
> and `talosctl logs -k` work without a running API server and are the primary diagnostic
> tools in this scenario.

## Complete Example: Upgrading from Kubernetes 1.34.2 to 1.35.0

This section walks through a concrete upgrade for a single-node cluster with the following parameters:

* Cluster context: `amiya.akn`
* Current Kubernetes version: `1.34.2`
* Current Talos version: `v1.13.0`
* Target Kubernetes version: `1.35.0`

### Step 1 — Verify current state

```sh
$ kubectl version
Client Version: v1.35.0
Server Version: v1.34.2

$ talosctl version --context amiya.akn
Server:
  NODE              VERSION
  10.0.0.10         v1.13.0

$ kubectl get nodes -o wide
NAME          STATUS   ROLES           AGE   VERSION   INTERNAL-IP
amiya-akn-01  Ready    control-plane   46d   v1.34.2   10.0.0.10
```

Talos v1.13.0 supports Kubernetes 1.34.x through 1.35.x — upgrading to 1.35.0 is within the supported range.

### Step 2 — Dry-run

```sh
$ talosctl --context amiya.akn upgrade-k8s --to 1.35.0 --dry-run
validating connection to nodes: ["10.0.0.10"]
discovered control plane node "10.0.0.10"

 > phase 1: pre-pulling images
   * "10.0.0.10": pulling image "registry.k8s.io/kube-apiserver:v1.35.0"
   * "10.0.0.10": pulling image "registry.k8s.io/kube-controller-manager:v1.35.0"
   * "10.0.0.10": pulling image "registry.k8s.io/kube-scheduler:v1.35.0"
   * "10.0.0.10": pulling image "registry.k8s.io/kube-proxy:v1.35.0"
   * "10.0.0.10": pulling image "registry.k8s.io/pause:3.9"
   * "10.0.0.10": pulling image "registry.k8s.io/coredns/coredns:v1.11.1"
 > phase 2: updating control plane
   (dry-run) patch machine config on "10.0.0.10"
 ...
DRY RUN: no changes applied
```

### Step 3 — Execute the upgrade

```sh
$ talosctl --context amiya.akn upgrade-k8s --to 1.35.0
validating connection to nodes: ["10.0.0.10"]
discovered control plane node "10.0.0.10"

 > phase 1: pre-pulling images
   * "10.0.0.10": pre-pulled image "registry.k8s.io/kube-apiserver:v1.35.0"
   * "10.0.0.10": pre-pulled image "registry.k8s.io/kube-controller-manager:v1.35.0"
   * "10.0.0.10": pre-pulled image "registry.k8s.io/kube-scheduler:v1.35.0"
   * "10.0.0.10": pre-pulled image "registry.k8s.io/kube-proxy:v1.35.0"
 > phase 2: updating control plane
   * "10.0.0.10": patching machine config
 > phase 3: updating kube-proxy
 > phase 4: updating kubelet
   * "10.0.0.10": kubelet updated to v1.35.0
 > phase 5: updating manifests
 > phase 6: pruning resources
upgrade to Kubernetes 1.35.0 complete
```

### Step 4 — Post-upgrade verification

```sh
$ kubectl get nodes -o wide
NAME          STATUS   ROLES           AGE   VERSION   INTERNAL-IP
amiya-akn-01  Ready    control-plane   46d   v1.35.0   10.0.0.10

$ kubectl version
Client Version: v1.35.0
Server Version: v1.35.0

$ kubectl get pods -n kube-system
NAME                                    READY   STATUS    RESTARTS   AGE
coredns-5d78c9869d-xxxxx                1/1     Running   0          3m
kube-apiserver-amiya-akn-01             1/1     Running   0          3m
kube-controller-manager-amiya-akn-01   1/1     Running   0          3m
kube-proxy-xxxxx                        1/1     Running   0          3m
kube-scheduler-amiya-akn-01             1/1     Running   0          3m
```

## Quick verifications

* **Node version**: `kubectl get nodes` — `VERSION` column must show `v<TARGET_K8S_VERSION>`.
* **API server version**: `kubectl version` — `Server Version` must match the target.
* **System pods**: `kubectl get pods -n kube-system` — all pods `Running` or `Completed`, no `CrashLoopBackOff`.
* **Kubelet version per node**: `kubectl get nodes -o custom-columns='NAME:.metadata.name,KUBELET:.status.nodeInfo.kubeletVersion'`

> \[!TIP]
> **Re-run safety**
>
> `talosctl upgrade-k8s` is idempotent. If any step fails or the command exits unexpectedly, simply re-run the same command. It will detect completed phases and pick up from where it left off.
