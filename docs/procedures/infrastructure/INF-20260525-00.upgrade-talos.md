## Procedure INF-20260525-00: Upgrading Talos OS on a Single-Node Cluster

This procedure details how to upgrade the Talos OS version on a single-node cluster managed by this repository. It covers the full lifecycle: pre-upgrade checks, image pre-pull, upgrade execution, post-upgrade verification, and rollback if the upgrade fails.

This procedure is safe to run on a **single-node cluster** only. There are no quorum concerns and no need for cordon/drain sequences, but workloads will be briefly unavailable during the node reboot.

### Technical framework and conventions

All commands are executed through the `mise` environment, which sets `TALOSCONFIG` and `KUBECONFIG` automatically. The `talosctl` binary used here is the repository wrapper located in `scripts/talosctl` — it is already on `PATH` after `mise install`.

The upgrade mechanism works by replacing the initramfs/installer image on the node. Talos verifies the new image, applies it to the inactive boot partition, reboots into the new version, and makes the boot change permanent only after a successful boot sequence. If the node fails to come back healthy, the bootloader falls back to the previous partition and a manual rollback can be triggered.

Key constraints for this repository:

* **One minor version at a time**: the Talos upgrade path requires going through each intermediate minor version (e.g., `v1.11.x` → `v1.12.x` → `v1.13.x`). Do not skip minor versions.
* **Patch versions within a minor can be jumped**: upgrading from `v1.12.0` to `v1.12.5` in one step is safe.
* **v1.9 network interface names**: Talos v1.9 replaced `eudev` with `systemd-udev`. Predictable network interface names may change. Verify interface names in machine config patches before upgrading to v1.9+.

### Prerequisites

Before starting, ensure the following tools are installed and accessible via the `mise` environment: `talosctl`, `kubectl`.

You must also have:

* Network access to the node (IP reachable from your workstation).
* `TALOSCONFIG` pointing to the correct endpoint — verify with `talosctl version`.
* A recent backup of the machine configuration (this procedure includes an export step).
* Confirmed that the target Talos version is compatible with the currently running Kubernetes version. Consult the [Talos support matrix](https://www.talos.dev/v1.13/introduction/support-matrix/).

### Required inputs

* `CLUSTER`: The talosconfig context name for the cluster (e.g., `amiya.akn`).
* `TARGET_VERSION`: Target Talos installer image tag (e.g., `v1.13.0`).

## Step 1 — Verify current state and upgrade path

Confirm the running version, check the upgrade path, and export the machine configuration as a backup.

```sh
# Check running Talos and Kubernetes versions
talosctl version --context <CLUSTER>
kubectl get nodes -o wide

# Export current machine configuration to a timestamped backup file
talosctl get machineconfig --context <CLUSTER> -o yaml > talos-machineconfig-backup-$(date +%Y%m%d).yaml
```

If you need the node IP (e.g., for an `apply-config --insecure` operation later), retrieve it from the context rather than hardcoding it:

```sh
# List all contexts with their configured endpoints and nodes
talosctl config contexts

# Or list cluster members (shows IP, hostname, role)
talosctl --context <CLUSTER> get members
```

> \[!NOTE]
> `talosctl config info --context <CLUSTER>` does not work with the `scripts/talosctl` wrapper in this repo — the wrapper intercepts all `config` subcommands. Use `talosctl config contexts` or `get members` instead.

> \[!IMPORTANT]
> **Upgrade path validation**
>
> You MUST upgrade through each intermediate minor version if you are crossing minor boundaries. For example, upgrading from v1.11.x to v1.13.x requires two separate upgrade operations:
>
> 1. v1.11.x → latest v1.12.x patch
> 2. v1.12.x → target v1.13.x
>
> Check the [Talos support matrix](https://www.talos.dev/v1.13/introduction/support-matrix/) to confirm which Kubernetes versions are supported by the target Talos version before proceeding.

> \[!NOTE]
> **v1.9 network interface name change**
>
> Talos v1.9 replaces `eudev` with `systemd-udev`. If your machine config patches reference network interface names by their kernel-assigned names (e.g., `eth0`, `enp3s0`), verify that the new udev rules will produce the same names on your hardware before upgrading.

## Step 2 — Pre-pull the installer image

Pre-pulling the installer image before the upgrade reduces the time the node spends offline during the upgrade window. This step is optional but strongly recommended.

```sh
# Pre-pull the target installer image
talosctl image pull --context <CLUSTER> \
  ghcr.io/siderolabs/installer:<TARGET_VERSION>
```

> \[!TIP]
> If the node has limited bandwidth or a slow connection to `ghcr.io`, pre-pulling the image at a low-traffic time significantly reduces the reboot window duration.

## Step 3 — Execute the upgrade

Run the upgrade command. The `--wait` flag causes `talosctl` to block until the node has rebooted and returned to a healthy state. The `--debug` flag streams verbose output including boot logs, which is useful on single-node clusters where there is no other node to observe from.

```sh
talosctl upgrade --context <CLUSTER> \
  --image ghcr.io/siderolabs/installer:<TARGET_VERSION> \
  --wait --debug
```

> \[!IMPORTANT]
> **Single-node workload interruption**
>
> The node will reboot as part of the upgrade. All workloads running on the node will be unavailable during the reboot window (typically 1–3 minutes). Plan the upgrade during a maintenance window if any services are externally exposed.

### Alternative: staged upgrade

If the standard upgrade fails because the node reports open file handles or locked resources, use the `--stage` flag. A staged upgrade reboots the node twice: once to stage the new installer, and a second time to boot into the new version.

```sh
talosctl upgrade --context <CLUSTER> --stage \
  --image ghcr.io/siderolabs/installer:<TARGET_VERSION>
```

> \[!NOTE]
> With `--stage`, the first reboot applies the staged image to the inactive boot partition. The second reboot switches to the new partition. The total offline window is longer than a standard upgrade.

## Step 4 — Monitor the upgrade

In a separate terminal, stream the kernel log from the node to observe the boot sequence in real time.

```sh
# Stream kernel messages during upgrade (run in a separate terminal)
talosctl dmesg --context <CLUSTER> -f
```

If `--wait --debug` was used in Step 3, progress is already streamed in the upgrade terminal. The `dmesg` stream is an additional diagnostic channel useful when the upgrade terminal loses connection mid-reboot (expected behavior — reconnects automatically).

The upgrade is complete when:

* `talosctl` reports the node is healthy, OR
* `kubectl get nodes` shows the node in `Ready` state.

## Step 5 — Post-upgrade verification

Confirm the node is running the expected Talos version and that the Kubernetes control plane is healthy.

```sh
# Verify Talos version
talosctl version --context <CLUSTER>

# Verify node is Ready in Kubernetes
kubectl get nodes -o wide

# Verify control plane pods are running
kubectl get pods -n kube-system

# Verify etcd health (single-node: one member, healthy)
talosctl etcd members --context <CLUSTER>
```

Expected outcome:

* `talosctl version` shows `<TARGET_VERSION>` under `Tag`.
* `kubectl get nodes` shows `Ready` with no taints other than the standard control-plane taint.
* All `kube-system` pods are `Running` or `Completed`.

## Step 6 — Update the bootstrap patch-config

After a successful Talos upgrade, update the `machine.install.image` field in the cluster's bootstrap patch-config to match the new version. This ensures that future re-bootstraps of the node use the correct installer image.

The field to update is:

```yaml
machine:
  install:
    image: factory.talos.dev/[installer-variant]/<SCHEMATIC_ID>:<TARGET_VERSION>
```

Locate the patch-config for the cluster:

* For `amiya.akn`: `projects/amiya.akn/docs/bootstrap/talos/<node>.patch-config.yaml`
* For `lungmen.akn`: `projects/lungmen.akn/src/infrastructure/talos/<node>.patch-config.yaml`

Find the `machine.install.image` line and update the version tag from the old version to `<TARGET_VERSION>`. Then commit the change.

For example, after upgrading `amiya.akn` from `v1.12.3` to `v1.13.0`:

```yaml
# Before (in projects/amiya.akn/docs/bootstrap/talos/amiya-akn-01.patch-config.yaml)
machine:
  install:
    image: factory.talos.dev/metal-installer-secureboot/<SCHEMATIC_ID>:v1.12.3

# After
machine:
  install:
    image: factory.talos.dev/metal-installer-secureboot/<SCHEMATIC_ID>:v1.13.0
```

## Step 7 — Rollback (if the upgrade fails)

If the node does not come back healthy after the upgrade, or if critical regressions are detected, roll back to the previous Talos version. Talos keeps the previous boot partition intact until the new version is confirmed.

```sh
talosctl rollback --context <CLUSTER>
```

> \[!IMPORTANT]
> **Rollback window**
>
> The rollback command is only effective before Talos makes the new boot partition permanent. This happens automatically after a successful boot sequence. If the node has already been confirmed healthy and workloads are running, the previous partition is no longer guaranteed to be intact.
>
> Run the rollback immediately after detecting a failure — do not wait.

After rollback, verify the node is back on the previous version:

```sh
talosctl version --context <CLUSTER>
kubectl get nodes -o wide
```

## Complete Example: Upgrading from v1.12.3 to v1.13.0

This section walks through a concrete upgrade for a single-node cluster with the following parameters:

* Cluster context: `amiya.akn`
* Current Talos version: `v1.12.3`
* Target Talos version: `v1.13.0`

### Step 1 — Verify current state

```sh
$ talosctl version --context amiya.akn
Client:
  Tag: v1.13.0        # talosctl client version (from mise)
Server:
  NODE              VERSION
  10.0.0.10         v1.12.3

$ kubectl get nodes -o wide
NAME          STATUS   ROLES           AGE   VERSION    INTERNAL-IP   ...
amiya-akn-01  Ready    control-plane   45d   v1.33.1    10.0.0.10     ...

# Export machine config backup
$ talosctl get machineconfig --context amiya.akn -o yaml > talos-machineconfig-backup-20260525.yaml

# Retrieve node IP from context if needed
$ talosctl config info --context amiya.akn
Context:         amiya.akn
Endpoints:       10.0.0.10
Nodes:           10.0.0.10
```

### Step 2 — Pre-pull the installer image

```sh
$ talosctl image pull --context amiya.akn \
    ghcr.io/siderolabs/installer:v1.13.0
pulling ghcr.io/siderolabs/installer:v1.13.0 ... done
```

### Step 3 — Execute the upgrade

```sh
$ talosctl upgrade --context amiya.akn \
    --image ghcr.io/siderolabs/installer:v1.13.0 \
    --wait --debug
watching for events...
amiya.akn: upgrade started, stage: "Upgrading"
amiya.akn: restarting into upgrade
... (node reboots, connection drops, reconnects automatically) ...
amiya.akn: upgrade complete
```

### Step 4 — Post-upgrade verification

```sh
$ talosctl version --context amiya.akn
Server:
  NODE              VERSION
  10.0.0.10         v1.13.0

$ kubectl get nodes
NAME          STATUS   ROLES           AGE   VERSION
amiya-akn-01  Ready    control-plane   45d   v1.33.1

$ kubectl get pods -n kube-system
NAME                                    READY   STATUS    RESTARTS   AGE
coredns-5d78c9869d-xxxxx                1/1     Running   0          2m
kube-apiserver-amiya-akn-01             1/1     Running   0          2m
kube-controller-manager-amiya-akn-01   1/1     Running   0          2m
kube-scheduler-amiya-akn-01             1/1     Running   0          2m
```

### Step 5 — Update the bootstrap patch-config

```sh
# Edit the patch-config to update the version tag
$ vim projects/amiya.akn/docs/bootstrap/talos/amiya-akn-01.patch-config.yaml
# Change: factory.talos.dev/metal-installer-secureboot/<SCHEMATIC_ID>:v1.12.3
# To:     factory.talos.dev/metal-installer-secureboot/<SCHEMATIC_ID>:v1.13.0

# Commit the change
$ git add projects/amiya.akn/docs/bootstrap/talos/amiya-akn-01.patch-config.yaml
$ git commit -S -m "~[project:amiya.akn]: Update Talos installer image to v1.13.0"
```

## Quick verifications

* **Talos version**: `talosctl version --context <CLUSTER>` — `Tag` field must match `<TARGET_VERSION>`.
* **Node readiness**: `kubectl get nodes` — node must be in `Ready` state.
* **System pods**: `kubectl get pods -n kube-system` — all pods `Running` or `Completed`, no `CrashLoopBackOff`.
* **Etcd health**: `talosctl etcd members --context <CLUSTER>` — single member, status `started`.

> \[!TIP]
> **Bootloader confirmation delay**
>
> Talos makes the new boot partition permanent only after the node has been running stably for a short period post-reboot. If you need to roll back, do so before running workloads for an extended period.
