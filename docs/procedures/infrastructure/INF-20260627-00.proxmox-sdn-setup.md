## Procedure INF-20260627-00: Provisioning the Proxmox SDN (simple zone + VNet for Talos clusters)

This procedure provisions the Proxmox VE SDN that underpins the dual-NIC Talos-on-Proxmox
design (ADR-014, V2). It creates one SDN zone, one shared VNet (`talosnet`) with its subnet,
enables gateway/SNAT/DHCP on it, and grants the `omni@pve` infra provider the `SDN.Use`
privilege it needs to attach Talos VM NICs to the VNet.

The result is a single shared, L2-isolated node-management network (`eth1`, `talosnet`) for
all Talos/Omni clusters — `lungmen.akn`, `rhodes.akn`, the sandbox, and any future cluster.
Cilium L2 LoadBalancer ARP announcements stay on `eth0` (`vmbr1`, VLAN 5) and are untouched by
this procedure.

This procedure is **additive and non-disruptive**. It creates new SDN objects and ACLs only; it
does not modify existing bridges, running VMs, or the existing `vmbr1` ACL. Existing Talos VMs
on VLAN 5 continue to work until they are migrated to the dual-NIC design.

> \[!NOTE]
> **Why a single shared VNet?** The original design allocated one VNet per cluster
> (`vnet-lungmen`, `vnet-rhodes`, `vnet-sandbox`). Implementation of
> [#1038](https://github.com/chezmoidotsh/arcane/issues/1038) hit a hard Omni constraint: a
> cluster-template's `patches[]` are Talos machine-config strategic-merge patches only and
> **cannot override a `MachineClass`'s `providerdata`** (where `additional_nics[].bridge`
> lives), and the `machineClass` reference exposes only `name` and `size`
> ([siderolabs/omni#2593](https://github.com/siderolabs/omni/issues/2593)). Per-cluster VNets
> would therefore force one MachineClass set per cluster, so all Talos clusters now share a
> single `talosnet`. Per-cluster isolation of *external* LoadBalancer traffic is unaffected —
> it comes from the per-cluster `/29` pools on VLAN 5 (`eth0`), not from the SDN VNet. See the
> ADR-014 Revision note (2026-06-27) for the full rationale and trade-offs.

> \[!IMPORTANT]
> **Authoritative IP plan.** Every VNet name, subnet, gateway, and option below comes from
> [`docs/network/vlans.md`](../network/vlans.md) (section "Proxmox SDN — VXLAN VNets"). If the
> values here ever disagree with that file, `vlans.md` wins — update this runbook to match.

### Technical framework and conventions

All commands run on the Proxmox node as `root`, either at the console or over SSH:

```sh
ssh root@pve-01.pve.chezmoi.sh
```

Two CLI tools drive the work:

* **`pvesh`** — the Proxmox API shell. In **PVE 9**, all SDN resources moved under
  `/cluster/sdn/` (not the legacy `/sdn/` path used in PVE 8). SDN changes are **staged**, then
  applied in one shot (Step 6). Think of it like a git index: nothing is live until you apply.
* **`pveum`** — the Proxmox user/ACL manager, used to grant `omni@pve` the `SDN.Use` role on the
  VNet so the Omni infra provider can attach VM NICs to it.

The subnet resource path is nested under its VNet: `/cluster/sdn/vnets/<vnet>/subnets`. There is
no top-level `/cluster/sdn/subnets` endpoint — always scope a subnet to its VNet.

> \[!NOTE]
> **PVE 9 SDN naming constraints.** Zone names are limited to **8 alphanumeric characters**.
> VNet names are limited to **8 alphanumeric characters** with no hyphens. These constraints
> drove the choices `pvenet` (zone) and `talosnet` (VNet) instead of the originally planned
> `localnetwork` / `vnet-talos`. The zone was initially named `localnet` but later renamed to
> `pvenet` to avoid confusion with the pre-existing `localnetwork` PVE ACL path.

> \[!NOTE]
> **Zone type — `simple` instead of `vxlan` (single-node decision).** The original design called
> for a `vxlan` zone to baseline multi-node operation. PVE 9 enforces that VXLAN zones require
> either `--peers <ip-list>` (static remote peer IPs) or `--fabric <fabric-id>` at creation time
> — making it impossible to create a VXLAN zone on a single-node homelab with no remote peers.
> A `simple` zone was used instead. When a second Proxmox node joins, migrate to VXLAN using the
> [Migrating to VXLAN](#migrating-from-simple-to-vxlan-when-a-second-proxmox-node-joins)
> procedure at the end of this document — it is non-disruptive to running VMs.

> \[!NOTE]
> **DHCP behavior in PVE 9 SDN.** The `dnsmasq@<zone>` service handles DHCP per zone. The
> generated config uses `dhcp-ignore=tag:!known` and a `static` range, which look like they
> would block dynamic assignment — but PVE's IPAM integration handles this transparently: when
> a VM or LXC NIC is configured with `ip=dhcp` on an SDN VNet, PVE automatically allocates an
> IP from the subnet pool, writes the `MAC,IP` entry into `ethers`, and reloads dnsmasq before
> the guest even sends its first DHCP DISCOVER. No manual reservation or config override is
> needed.

### Prerequisites

Before starting, ensure:

* **PVE 8.x or newer** with the SDN plugin enabled (Datacenter → SDN is visible in the UI). SDN
  ships with PVE 8; no extra package is needed.
* **`dnsmasq` package installed.** The SDN DHCP feature requires it:
  ```sh
  apt-get install -y dnsmasq
  systemctl disable dnsmasq.service   # disable the bare service; SDN uses dnsmasq@<zone> instances
  ```
* **Root access** to `pve-01` (SSH key-based, per the Omni provider prerequisites).
* **The `omni@pve` user, `talos` pool, and existing ACLs already exist** — created by the Omni
  infra provider setup, including the bridge-scoped ACL at
  `/sdn/zones/localnetwork/vmbr1` (see
  [Proxmox user and role setup](../../projects/chezmoi.sh/src/infrastructure/proxmox/lxc/omni-infra-provider-proxmox/README.md#proxmox-user-and-role-setup)).
  This procedure *extends* that ACL; it does not recreate it.
* You have read [ADR-014](../decisions/014-network-topology.md) and the SDN table in
  [`vlans.md`](../network/vlans.md) and understand the dual-NIC design (`eth0` = Cilium L2 on
  VLAN 5, `eth1` = node traffic on the SDN VNet).

## Step 1 — Verify or create the SDN zone

The `pvenet` zone was not created by prior tooling — it must be created here. First check
whether it already exists:

```sh
pvesh get /cluster/sdn/zones/pvenet
# Expected: type = simple, ipam = pve, dhcp = dnsmasq
```

**If the zone already exists with the correct attributes:** skip to Step 2.

**If the zone does not exist** (normal for a fresh setup): create it:

```sh
pvesh create /cluster/sdn/zones \
  --type simple \
  --zone pvenet \
  --ipam pve
```

Then enable DHCP on the zone (required for the `dnsmasq@pvenet` service to start):

```sh
pvesh set /cluster/sdn/zones/pvenet --dhcp dnsmasq
```

Verify the zone is staged:

```sh
pvesh get /cluster/sdn/zones
# → one entry: zone = pvenet, type = simple, ipam = pve, dhcp = dnsmasq
```

***

## Step 2 — Create the VNet

Create the single shared VNet `talosnet`, assigned to the `pvenet` zone. The VNet becomes
its own isolated Linux bridge named `talosnet`, which is the bridge name you attach a VM NIC to.

```sh
pvesh create /cluster/sdn/vnets \
  --type vnet \
  --vnet talosnet \
  --zone pvenet \
  --alias "Shared Talos node traffic (eth1) for all clusters"
```

Verify the VNet is staged:

```sh
pvesh get /cluster/sdn/vnets
# → talosnet — zone = pvenet
```

***

## Step 3 — Create the subnet (gateway + SNAT + DHCP range)

Create one subnet scoped under `talosnet`. The subnet sets:

* **gateway `10.128.0.1`** — the PVE node acts as the L3 router for the VNet.
* **`snat 1`** — Source NAT. This is **critical for CSI**: it is the only way a node (which has
  no VLAN 5 IP) can reach the Proxmox API at `pve-01:8006`, which `proxmox-csi-plugin` requires
  to attach and detach volumes.
* **`dhcp-range`** — the IPAM pool (`10.128.0.10`–`10.128.0.250`). PVE allocates from this
  range and writes a `MAC,IP` reservation into `ethers` automatically when a NIC is configured
  with `ip=dhcp` on the VNet.

```sh
pvesh create /cluster/sdn/vnets/talosnet/subnets \
  --type subnet \
  --subnet 10.128.0.0/24 \
  --gateway 10.128.0.1 \
  --snat 1 \
  --dhcp-range start-address=10.128.0.10,end-address=10.128.0.250
```

> \[!IMPORTANT]
> **Do not skip `snat 1`.** Without SNAT, Talos nodes cannot reach the Proxmox API and
> `proxmox-csi-plugin` fails to mount/detach volumes. This is a hard requirement of ADR-014.

Verify the subnet is staged under the VNet:

```sh
pvesh get /cluster/sdn/vnets/talosnet/subnets
# → one subnet: 10.128.0.0/24, gateway 10.128.0.1, snat=1, dhcp 10.128.0.10-10.128.0.250
```

***

## Step 4 — Verify DHCP is operational

PVE manages DHCP transparently through its IPAM integration. When a VM or LXC NIC is
configured with `ip=dhcp` on `talosnet`, PVE:

1. Allocates an IP from the `10.128.0.10–10.128.0.250` pool
2. Writes `MAC,IP` into `/etc/dnsmasq.d/pvenet/ethers`
3. Signals `dnsmasq@pvenet` to reload before the guest's first DHCP DISCOVER

No manual reservation is required.

**Verify the DHCP service is active:**

```sh
systemctl is-active dnsmasq@pvenet   # → active

# After a VM/LXC boots with ip=dhcp on talosnet, verify the lease appeared:
cat /var/lib/misc/dnsmasq.pvenet.leases   # MAC <-> IP currently held
cat /etc/dnsmasq.d/pvenet/ethers          # IPAM-generated MAC → IP reservations
```

> \[!NOTE]
> To pin a specific IP to a MAC (e.g. for a Talos node that must always get the same
> address regardless of IPAM allocation order), register an explicit reservation:
>
> ```sh
> pvesh create /cluster/sdn/vnets/talosnet/ips \
>   --ip 10.128.0.XX \
>   --mac AA:BB:CC:DD:EE:FF \
>   --zone pvenet \
>   --subnet 10.128.0.0/24
> ```
>
> Or via the UI: Datacenter → SDN → `pvenet` → `talosnet` → subnet → IP Management.

## Step 5 — Grant `omni@pve` the SDN ACL on the VNet

The Omni infra provider attaches each Talos VM's `eth1` to the `talosnet` bridge. Attaching a
NIC to an SDN object requires the `SDN.Use` privilege, which the built-in `PVESDNUser` role
grants.

The existing setup already grants `SDN.Use` on the `vmbr1` bridge only:

```sh
pvesh get /access/acl | grep -i sdn
# → /sdn/zones/localnetwork/vmbr1  user omni@pve  role PVESDNUser   (existing — keep it)
```

That ACL covers `vmbr1` (the `eth0` Cilium-L2 plane) but **not** `talosnet`. Grant the role on
the VNet — this *extends* the existing ACL, it does not replace it:

```sh
pveum acl modify /sdn/zones/pvenet/talosnet --users omni@pve --roles PVESDNUser
```

> \[!NOTE]
> **Keep the `vmbr1` ACL.** The dual-NIC design still attaches `eth0` to `vmbr1` for Cilium L2
> announcements. Do not remove the original `/sdn/zones/localnetwork/vmbr1` ACL — the VNet ACL is
> *in addition to* it.

Verify both SDN ACLs are present:

```sh
pvesh get /access/acl | grep -i 'sdn/zones'
# → /sdn/zones/localnetwork/vmbr1  (existing, eth0 plane)
# → /sdn/zones/pvenet/talosnet     (new, eth1 plane)
# both: omni@pve / PVESDNUser
```

## Step 6 — Apply the SDN configuration and verify

SDN changes are inert until applied. In PVE 9, the apply command is `pvesh set /cluster/sdn`
(not `pvesh create /sdn` as in PVE 8). This triggers a full network reload via `ifreload -a`
and starts the `dnsmasq@pvenet` DHCP service.

```sh
pvesh set /cluster/sdn
```

> \[!IMPORTANT]
> **`source-directory` directive required.** PVE 9 writes the SDN bridge configuration to
> `/etc/network/interfaces.d/sdn`, but `ifupdown2` will not read it unless the main
> `/etc/network/interfaces` file includes a `source-directory` directive. If the `talosnet`
> bridge does not come up after the apply, add the directive and reload:
>
> ```sh
> # Check if the directive is present:
> grep "source-directory" /etc/network/interfaces
>
> # If missing, add it (PVE preserves source/source-directory directives):
> sed -i '/^auto lo/i source-directory /etc/network/interfaces.d/\n' /etc/network/interfaces
>
> # Reload to bring up talosnet:
> pvesh set /cluster/sdn
> ```

Then verify the full state is live.

**Zone, VNet, subnet:**

```sh
pvesh get /cluster/sdn/zones/pvenet
pvesh get /cluster/sdn/vnets/talosnet
pvesh get /cluster/sdn/vnets/talosnet/subnets
```

**Bridge is up on the node:**

```sh
ip -br link show talosnet
# → talosnet  UP  (or UNKNOWN — normal for a bridge with no ports)
ip addr show talosnet | grep "inet "
# → inet 10.128.0.1/24
```

**SNAT reachability to the Proxmox API (the CSI requirement).** From a VM whose `eth1` is on the
VNet, the node must reach the Proxmox API through the SNAT gateway:

```sh
# From inside a Talos VM (or any guest on the VNet):
curl -k https://10.0.0.11:8006/api2/json/version
# → JSON with {"version":"...","release":"..."}  → SNAT + API reachability confirmed
```

If you do not yet have a VM on the VNet, you can sanity-check the gateway/SNAT plumbing from the
node itself:

```sh
ip addr show talosnet                          # → the .1 gateway is on the node
iptables -t nat -L POSTROUTING -n | grep 10.128.0   # → SNAT rule for 10.128.0.0/24 to 10.0.0.11
```

**DHCP service running:**

```sh
systemctl is-active dnsmasq@pvenet
# → active
pgrep -a dnsmasq | grep pvenet
```

* **Gateway**: `ip addr show talosnet | grep inet` → `10.128.0.1/24`.

## Notes and troubleshooting

### SDN apply did not bring the bridge up

If `ip -br link show talosnet` is empty after `pvesh set /cluster/sdn`:

```sh
# 1. Check the source-directory directive is in /etc/network/interfaces
grep "source-directory" /etc/network/interfaces
# If missing:
sed -i '/^auto lo/i source-directory /etc/network/interfaces.d/\n' /etc/network/interfaces

# 2. Check the SDN-generated interface config
cat /etc/network/interfaces.d/sdn

# 3. Re-apply
pvesh set /cluster/sdn

# 4. If still not up, bring it up directly:
ifup talosnet

# 5. Check journalctl for errors
journalctl -u 'pve*' -n 50 --no-pager
```

### VM provisioning fails with 403 on NIC attach

The provider lacks `SDN.Use` on the VNet. Confirm the ACL from Step 5 exists:

```sh
pvesh get /access/acl | grep talosnet
```

If missing, re-run the `pveum acl modify` from Step 5.

### CSI cannot reach the Proxmox API (SNAT)

`snat 1` must be set on the subnet (Step 3) and the SDN applied (Step 6). Verify:

```sh
iptables -t nat -L POSTROUTING -n | grep 10.128.0
# → must show one SNAT rule to 10.0.0.11

curl -k https://10.0.0.11:8006/api2/json/version   # from a guest on talosnet
```

A missing rule means SNAT is not active — re-check the subnet's `snat` flag and re-apply.

### dnsmasq\@pvenet fails to start

If `systemctl status dnsmasq@pvenet` shows "Address already in use":

```sh
# The bare dnsmasq.service conflicts with the zone instance. Stop and disable it:
systemctl stop dnsmasq.service
systemctl disable dnsmasq.service

# Then restart the zone instance:
systemctl start dnsmasq@pvenet
systemctl is-active dnsmasq@pvenet
```

### Node did not get a DHCP address

PVE 9 DHCP is **explicit-reservation only** (`dhcp-ignore=tag:!known`). A node with an
unregistered MAC is silently ignored. Check:

```sh
cat /etc/dnsmasq.d/pvenet/ethers   # MAC → IP reservations
cat /var/lib/misc/dnsmasq.pvenet.leases  # active leases
```

If the node's MAC is not in `ethers`, register a reservation via the PVE IPAM (Step 4).

### Duplicate SNAT rules after ifdown/ifup cycle

If `iptables -t nat -L POSTROUTING -n | grep 10.128.0` shows two identical rules, remove one:

```sh
iptables -t nat -D POSTROUTING -s 10.128.0.0/24 -o vmbr0 -j SNAT --to-source 10.0.0.11
# Run once per extra copy. Verify only one rule remains.
```

## Migrating from simple to VXLAN (when a second Proxmox node joins)

When a second PVE node (`pve-02`) joins the cluster, migrate the `pvenet` zone from `simple`
to `vxlan`. **Running VMs are not disrupted** — the VXLAN migration is a zone-level operation
that only changes the encapsulation; existing bridges, subnets, and DHCP leases are preserved.

> \[!WARNING]
> This section is a design blueprint, not a tested runbook. Validate each step against the
> [PVE SDN wiki](https://pve.proxmox.com/wiki/Software-Defined_Network) before executing on a
> production node. In particular, verify:
>
> 1. Whether a VXLAN zone with `--peers` can be created **in place** on PVE 9, or whether the
>    zone must be deleted and recreated (VNets must be migrated separately if so).
> 2. Whether SNAT and DHCP (`dhcp: dnsmasq`) are supported on `vxlan`-type zones in PVE 9
>    (they are documented only for `simple` zones in the current PVE guide).
> 3. The exact VXLAN tunnel IP for each node (typically the management IP, e.g. `10.0.0.11`
>    for `pve-01`, `10.0.0.12` for `pve-02`).

### Conceptual migration sequence

```sh
# On the current single node (before pve-02 joins):
# 1. Note the current zone config:
pvesh get /cluster/sdn/zones/pvenet

# After pve-02 joins the PVE cluster:

# 2. Update the zone type to vxlan with both node IPs as peers.
#    (Or delete + recreate if in-place zone type change is unsupported.)
pvesh set /cluster/sdn/zones/pvenet \
  --type vxlan \
  --peers 10.0.0.11,10.0.0.12

# 3. Verify VNet, subnet, DHCP, and SNAT settings are preserved:
pvesh get /cluster/sdn/vnets/talosnet/subnets

# 4. Apply:
pvesh set /cluster/sdn

# 5. Verify bridge is up on both nodes:
ssh root@pve-01.pve.chezmoi.sh ip -br link show talosnet
ssh root@pve-02.pve.chezmoi.sh ip -br link show talosnet
```

The Talos VM NICs (`eth1` → `talosnet`) and the Omni ACL (`/sdn/zones/pvenet/talosnet`) do
not change — only the zone's encapsulation type changes.

PVE 9 deviations from the prior design: (1) SDN API moved to `/cluster/sdn/`; (2) zone names limited to 8 chars → `localnetwork` → `localnet`; (3) VNet names limited to 8 alphanumeric chars (no hyphens) → `vnet-talos` → `talosnet`; (4) VXLAN zone requires `--peers` or `--fabric` at creation time — impossible on a single node with no remote peers → fell back to `simple` zone; (5) zone requires explicit `--dhcp dnsmasq` flag; (6) `dnsmasq` package must be installed and the bare `dnsmasq.service` must be disabled (conflicts with the per-zone `dnsmasq@localnet` instance); (7) `/etc/network/interfaces` needs a `source-directory /etc/network/interfaces.d/` directive to persist the SDN bridge across reboots; (8) PVE DHCP works through IPAM integration — `dhcp-ignore=tag:!known` and `static` range are generated but PVE auto-registers MAC→IP in `ethers` when a NIC is created with `ip=dhcp` on the VNet, so no manual reservation or config override is needed.
