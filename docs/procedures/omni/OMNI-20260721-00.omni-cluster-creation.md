# Procedure OMNI-20260721-00: Talos cluster bring-up on Proxmox (Omni provisioning through GitOps registration)

This procedure brings up a new Talos Linux cluster on Proxmox VE end to end: provisioning through Omni (Sidero Omni),
bootstrapping the CNI, provisioning the Proxmox CCM/CSI credentials, deploying the CCM/CSI plugins, registering the
cluster into this repository's GitOps setup (as the ArgoCD hub or as a spoke), and validating that networking, storage,
and cloud-provider integration actually work — not just that pods exist.

> \[!IMPORTANT] **This procedure only _consumes_ Omni** to provision a downstream cluster; it does not modify Omni's own
> hosting. See [Omni](#omni) below for where Omni itself runs, and
> [Hub \& spoke GitOps topology](#hub--spoke-gitops-topology) for how the new cluster's ArgoCD role (hub or spoke) is
> decided — the two are unrelated and must not be conflated.

## Overview

End to end, this procedure produces: a Talos cluster in the `Ready` state, with Cilium (CNI), the Proxmox CSI plugin,
and the Proxmox CCM all functional and independently verified — not merely "pods are Running" — and the cluster
registered into the repo's ArgoCD-based GitOps setup, either as the central hub (self-hosted ArgoCD) or as a spoke
registered against an existing hub. The steps are ordered deliberately: CNI is bootstrapped before anything else because
nothing in the cluster works without pod networking; the Proxmox credentials and CSI/CCM come next because storage and
node topology are needed by most workloads; GitOps registration comes last, once the cluster's own infrastructure is
functional and verified.

## Contextual information

### Omni

[Omni (Sidero Omni)](https://omni.siderolabs.com) is a declarative Talos cluster management plane: cluster templates
describe the desired state (Kubernetes/Talos versions, machine classes, CIDRs, bootstrap manifests), and Omni provisions
and reconciles Proxmox VMs against that state.

Omni itself runs as a **standalone Proxmox LXC**
([NixOS + Omni + a co-located Dex OIDC provider + Caddy](projects/chezmoi.sh/src/infrastructure/proxmox/lxc/omni/)) on
the [`pve-01.pve.chezmoi.sh` Proxmox host](projects/chezmoi.sh/docs/PROXMOX-VE.md). It is **not** a workload inside any
Kubernetes cluster. `OMNICONFIG` points to a local config file (set by this repo's `mise` config) and populated by
Omni's own login flow (`omnictl config new/add`).

> \[!IMPORTANT] **Do not conflate Omni's hosting with the ArgoCD hub.** Omni manages every cluster regardless of which
> one currently hosts the GitOps hub — the two are independent concerns (see
> [Hub \& spoke GitOps topology](#hub--spoke-gitops-topology)).

### Talos

[Talos Linux](https://www.talos.dev) is an immutable, API-driven Linux distribution purpose-built for Kubernetes: no
SSH, no shell, no package manager — configuration is applied exclusively through machine config patches (YAML) via the
Talos API.

### Hub & spoke GitOps topology

This repository's GitOps engine is ArgoCD, run as a single central instance that manages every cluster in the homelab (a
"hub and spoke" topology, not per-cluster ArgoCD installs). One cluster **self-hosts** ArgoCD and acts as the **hub** —
it self-registers as an ArgoCD cluster _(an `argocd.argoproj.io/secret-type: cluster` Secret, provisioned by the ArgoCD
Helm chart itself)_ so it manages its own workloads the same way it manages every other cluster. Every other cluster is
a **spoke**: it registers into the hub's ArgoCD via `argocd cluster add <name> --name <name>` and has no `argocd/`
directory of its own.

This leaves an explicit fork for this procedure's registration step (Step 11):

- **If the cluster being brought up is the hub** — self-host: apply its own ArgoCD manifests directly (the
  `seed.application.yaml` pattern).
- **If it is a spoke** — register it into the existing hub's ArgoCD with `argocd cluster add <name> --name <name>`.

Which branch applies is a per-cluster decision (see `GITOPS_ROLE` in [Required inputs](#required-inputs)) — this
procedure does not prescribe it.

### Technical framework and conventions

Omni manages Talos clusters through three independent building blocks:

- **Cluster templates** (`*.clustertemplate.yaml`) — Declarative YAML describing the desired cluster: name, Kubernetes
  and Talos versions, control-plane and worker machine classes, pod and service CIDRs, and bootstrap manifest URLs.
  These are **flat multi-doc YAML** with only a top-level `kind` and `name` per document — there is **no `apiVersion`,
  `metadata`, or `spec`** field. This format is **incompatible with kustomize** (which requires `metadata`), so
  templates are **standalone full copies edited in place**, never kustomize overlays.
- **Machine classes** (COSI resources) — Define VM shapes (vCPU, memory, disk) and Proxmox placement (bridges, VLANs,
  tags). They are **independent of cluster templates** and applied separately via `omnictl apply`. They live in
  `projects/chezmoi.sh/src/infrastructure/omni/machineclasses/`.
- **Bootstrap manifests** — Referenced by commit SHA in the template through `raw.githubusercontent.com` URLs. Cilium
  (the CNI) and metric-server are applied at bootstrap — see the note in [Step 3](#step-3--create-the-cluster-template)
  for why this bootstrap-time mechanism is used instead of GitOps. The pinned SHA **must be reachable on GitHub's `main`
  branch** at apply time, or the CNI is silently never installed (see [Known issues](#known-issues)).

`omnictl` is provisioned through `mise` (`mise install`). The `OMNICONFIG` environment variable points to a local config
file, populated by Omni's own login flow (`omnictl config new` for first-time setup, `omnictl config add` to add a
context).

### Prerequisites

Before starting, ensure:

- **Proxmox VE SDN is operational** — the shared `talosnet` VNet, its subnet, and the `SDN.Use` ACL for `omni@pve`
  exist, provisioned via the Pulumi `stack/proxmox` SDN config
  (`projects/chezmoi.sh/src/infrastructure/pulumi/stack/proxmox/sdn.ts`). Apply that stack first if not already done.
- **`omnictl` available** (`mise install`).
- **Omni authentication** — `OMNICONFIG` populated via Omni's own login flow (`omnictl config new` /
  `omnictl config add`).
- **Machine classes applied in Omni** — they are COSI resources, not part of the template. If Omni was reset, re-apply
  them (Step 1).
- **The pinned commit SHA is reachable on `main`** — see Step 4.
- **The Proxmox CCM/CSI token stack is applied** — `projects/chezmoi.sh/src/infrastructure/pulumi/stack/proxmox/`
  (`access.ts`) must already be applied so the `kubernetes-ccm@pve` / `kubernetes-csi@pve` tokens exist before Step 9.

> \[!WARNING] Step 9's per-cluster delivery code does not exist yet for any cluster — this procedure cannot be run fully
> end-to-end until it does. See Step 9's own TODO callout.

### Pod CIDR allocation

ADR-014 defines the pod CIDR supernet (`172.30.0.0/16`) and the principle that each cluster allocates a non-overlapping
`/19` from it. The authoritative per-cluster allocation table lives in
[`docs/network/vlans.md`](../../network/vlans.md). The next available CIDR is picked by incrementing the cluster ID:

| Cluster ID | Cluster  | Pod CIDR          |
| ---------- | -------- | ----------------- |
| 1          | rhodes   | `172.30.0.0/19`   |
| 2          | lungmen  | `172.30.32.0/19`  |
| 3          | _(next)_ | `172.30.64.0/19`  |
| …          | …        | …                 |
| 8          | sandbox  | `172.30.224.0/19` |

The **service CIDR `172.31.0.0/19`** and **clusterDNS `172.31.0.10`** are **shared across all clusters**
(ClusterMesh-ready). Do not change them.

## Step 1 — Apply machine classes

Machine classes are COSI resources applied via Omni, independent of the cluster template. They are shared infrastructure
defined in the `chezmoi.sh` project:

```sh
cd projects/chezmoi.sh
mise run omni:machineclass:diff     # dry-run / preview
mise run omni:machineclass:apply    # apply all machine classes to Omni
```

> \[!NOTE] Only needed if machine classes changed, or Omni was reset/is empty — otherwise this step is a no-op, safe to
> run regardless. Verify existing classes with `mise run omni:machineclass:list`.

Authentication is required first — `omnictl config new` (or `omnictl config add` if a context already exists) populates
`OMNICONFIG`.

---

## Step 2 — Allocate the cluster identity

Pick the next cluster ID and its corresponding pod CIDR from the [allocation table](#pod-cidr-allocation). For example,
with `rhodes` (ID 1) and `lungmen` (ID 2) taken, the next cluster is ID 3 → pod CIDR `172.30.64.0/19`.

Choose a cluster name (e.g. `rhodes-akn`) and a project directory (`projects/<cluster>.akn/`).

> \[!IMPORTANT] **Do not change the service CIDR or clusterDNS.** `serviceSubnets` must stay `172.31.0.0/19` and
> `clusterDNS` must stay `172.31.0.10`. Keeping them identical across every cluster — instead of allocating a
> per-cluster range the way the pod CIDR does — eliminates any risk of service-IP collisions once ClusterMesh connects
> clusters together, and a single shared `clusterDNS` value keeps things simple wherever DNS resolution has custom
> handling layered on top (e.g. a DNS setup that resolves into Tailscale). Only the pod CIDR (`podSubnets`) differs per
> cluster.

---

## Step 3 — Create the cluster template

Copy the reference base template into the cluster's project and override the fields in place:

```sh
cp catalog/omni/clustertemplates/base.yaml \
   projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml
```

Fields to override from the base:

- `kind: Cluster` → `name:` — the cluster name (e.g. `rhodes-akn`).
- `podSubnets:` — the cluster's allocated `/19` (e.g. `172.30.64.0/19`).
- `controlPlane.machineClass.size:` — number of control-plane nodes (usually `1` or `3`).
- `workers.machineClass.size:` — number of worker nodes.
- Verify `serviceSubnets` stays `172.31.0.0/19` and `clusterDNS` stays `172.31.0.10`.

> \[!IMPORTANT] **Templates cannot be kustomized.** Omni cluster templates are flat multi-doc YAML whose documents carry
> only a top-level `kind` and `name` — there is no `apiVersion`, `metadata`, or `spec`. Because kustomize requires
> `metadata`, these templates **cannot** be used as kustomize overlays. Each cluster gets a **full standalone copy** of
> the base template, edited in place. Do not attempt to share a template via `kustomization.yaml`.

> \[!NOTE] **Why the CNI is set here, not via GitOps.** Talos is immutable and API-driven — there's no `kubectl apply`
> available until the cluster already has pod networking, and GitOps (ArgoCD) itself needs a functioning cluster to sync
> anything into. Talos/Omni's `extraManifests` / `cni.urls` bootstrap mechanism (set in this template — see
> [Technical framework and conventions](#technical-framework-and-conventions)) is the **only** way to get a CNI running
> before the cluster has one, which is why Cilium is applied at Omni bootstrap time (Step 6/8) instead of through GitOps
> like every other component in this procedure.

---

## Step 4 — Verify SHA reachability

Before the template can bootstrap, the pinned commit SHA must exist on GitHub's `main` branch. After any branch is
squash-merged, confirm the URL resolves:

```sh
curl -sI "https://raw.githubusercontent.com/chezmoidotsh/arcane/<SHA>/catalog/talos/manifests/cilium/1.19.5-native.yaml" | head -1
# → HTTP/2 200
```

If the URL returns `404`, the SHA is not reachable (the branch tip was not preserved on `main` after squash-merge). It
must be repinned to the merge-commit SHA before proceeding — see
[OMNI-20260629-03 — SHA pinning](./OMNI-20260629-03.sha-repin.md).

> \[!WARNING] Applying a template whose pinned SHA 404s produces a cluster with **no CNI**. Cilium never downloads, pods
> stay in `ContainerCreating`, and the failure is silent. Always verify reachability here, before Step 6.

---

## Step 5 — Validate the cluster template

Run offline validation to confirm the template is well-formed:

```sh
omnictl cluster template validate -f projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml
```

Cluster projects are expected to wrap this in a mise task, e.g.:

```sh
mise run omni:clustertemplate:validate
```

> \[!NOTE] Each cluster project is expected to define this mise task in its own `.mise.toml` (mirroring
> `projects/lungmen.akn/`). Add it to the new project's `.mise.toml` if missing.

---

## Step 6 — Apply the cluster template

Once validated, apply the template to Omni. This step is online and requires authentication:

```sh
omnictl apply -f projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml
```

> \[!IMPORTANT] Use `omnictl apply -f`, **not** `omnictl cluster template sync`. Both the template file headers and
> `catalog/omni/README.md` specify `omnictl apply -f` as the application command. The two are not interchangeable:
> `sync` reconciles with full create/update/**delete** semantics against a directory of template files — it will delete
> any live resource not present in what it processes, and can touch multiple clusters in a single invocation if pointed
> at a directory containing more than one template. `apply -f` only creates/updates from a single named file and never
> deletes anything, which is what makes it the safe, correct command for applying one cluster's template here.

> \[!IMPORTANT] **The `kubelet.extraArgs.cloud-provider: external` machine config patch must be present in the template
> applied at this step.** The Proxmox CCM (Step 10) relies on it entirely — without it, the CCM silently skips node
> initialization (no error, no log). If the flag is missing and a node has already registered, fixing it afterwards
> requires **deleting and recreating the node**: Kubernetes' `providerID` field is immutable once set, so an in-place
> kubelet restart with the corrected flag does not retroactively fix an already-registered node. Source:
> `docs/experiments/20260617-proxmox-csi-ccm/README.md`, section "9.1 Known Considerations". Confirm the base template
> (or your copy of it) carries:
>
> ```yaml
> machine:
>   kubelet:
>     extraArgs:
>       cloud-provider: external
> ```

---

## Step 7 — Monitor cluster convergence

Omni begins provisioning VMs on Proxmox and bootstrapping Talos. Monitor convergence:

```sh
omnictl cluster status <cluster-name>
omnictl machine status
```

Convergence typically takes 5–15 minutes depending on VM startup time. Watch for all nodes to reach the `Ready` state
and for the control plane to form quorum.

> \[!NOTE] If convergence stalls beyond 15 minutes, check the Omni UI for machine-level errors. Common causes: Proxmox
> resource exhaustion, unreachable Talos endpoints, or missing machine classes (re-apply via Step 1).

---

## Step 8 — Retrieve the kubeconfig and verify CNI

Once the cluster shows `Ready`, download the kubeconfig and verify node and pod health:

```sh
export KUBECONFIG=~/.kube/config-<cluster-name>
omnictl kubeconfig --cluster <cluster-name> $KUBECONFIG

kubectl get nodes -o wide
kubectl get pods -A
```

Verify Cilium is running:

```sh
kubectl get pods -n kube-system | grep cilium
```

> \[!WARNING] **Known Cilium behaviour.** When Cilium starts in native (kube-proxy-replacement) mode, there is a brief
> (\~2 minute) window where node-to-node connectivity is interrupted while Cilium replaces kube-proxy. This is expected
> and self-resolves — do not intervene. If Cilium pods never appear at all, the pinned SHA was unreachable; see
> [Known issues](#known-issues) below.

## Step 11 — Register into GitOps (hub or spoke)

Apply the fork decided in [Hub \& spoke GitOps topology](#hub--spoke-gitops-topology):

## Validation checklist

Confirm CNI, CSI, and CCM are functioning, not just that pods exist. This adapts the validation criteria from
`docs/experiments/20260617-proxmox-csi-ccm/scripts/validate.sh` and
`docs/experiments/20260617-proxmox-csi-ccm/manifests/test-pvc.yaml` — deploy the latter (or an equivalent
`StorageClass`-targeted PVC) before running the CSI/CCM checks below.

> \[!NOTE] A dedicated validation script (mirroring `docs/experiments/20260617-proxmox-csi-ccm/scripts/validate.sh`)
> would be a good addition here; none exists yet for this procedure, so run the commands below manually until one does.

**CNI**

```sh
cilium status --wait
kubectl get pods -n kube-system | grep cilium   # all Running
```

**CCM / CSI** (criteria V-001 through V-010, from the experiment's validation table):

| ID    | Criterion                          | Command                                                                                        |
| ----- | ---------------------------------- | ---------------------------------------------------------------------------------------------- |
| V-001 | CCM pods running in `kube-system`  | `kubectl -n kube-system get pods -l app=proxmox-cloud-controller-manager`                      |
| V-002 | Node has topology labels           | `kubectl get nodes --show-labels \| grep topology`                                             |
| V-003 | Node has `providerID`              | `kubectl get nodes -o jsonpath='{.items[*].spec.providerID}'`                                  |
| V-004 | `CSIDriver` registered             | `kubectl get csidriver csi.proxmox.sinextra.dev`                                               |
| V-005 | CSI controller + node pods running | `kubectl -n kube-system get pods \| grep proxmox-csi`                                          |
| V-006 | `StorageClass` available           | `kubectl get sc proxmox-lvmthin-ext4` (or the storage class defined for this cluster)          |
| V-007 | Test PVC bound                     | `kubectl get pvc data-test-csi-0 \| grep Bound`                                                |
| V-008 | Test workload ready                | `kubectl get sts test-csi \| grep 1/1`                                                         |
| V-009 | Volume expansion works             | `kubectl patch pvc data-test-csi-0 -p '{"spec":{"resources":{"requests":{"storage":"2Gi"}}}}'` |
| V-010 | Volume survives pod restart        | Delete the pod, confirm the StatefulSet recreates it bound to the same PVC                     |

## Known issues

> \[!NOTE] This section documents real problems actually encountered while running this procedure — not hypothetical
> risks. Hit something not listed here? Add it. If the procedure itself needs correcting instead, update it directly.

### CNI auto-apply may silently fail

If the pinned SHA is unreachable (HTTP 404), Cilium never downloads and the cluster has no networking. Pods stay in
`ContainerCreating`. Mitigation: verify SHA reachability (Step 4) before applying. If it has already failed, repin the
SHA ([OMNI-20260629-03](./OMNI-20260629-03.sha-repin.md)) and re-apply the template.

### \~2-minute connectivity blip during Cilium kubeProxyReplacement takeover

When Cilium starts in native (kube-proxy-replacement) mode, there is a brief window where node-to-node connectivity is
interrupted while Cilium replaces kube-proxy. This is expected and self-resolves.

### Machine classes are erased on Omni reset

If the Omni instance is reset or reinstalled, all machine classes are lost. They must be re-applied
(`mise run omni:machineclass:apply` from `projects/chezmoi.sh/`) before any cluster template can converge. The cluster
templates themselves persist in Git.

### SHA pinning caveat

Always ensure the pinned SHA is reachable on `main` after a squash-merge. A branch-only SHA will 404 once the branch
commits are garbage-collected. See [OMNI-20260629-03 — SHA pinning](./OMNI-20260629-03.sha-repin.md).
