## Procedure OMNI-20260629-00: Omni cluster creation (Talos on Proxmox via Sidero Omni)

This procedure creates a new Talos Linux cluster managed by Omni (Sidero Omni) on Proxmox VE
virtual machines. Omni is a cluster management plane that declaratively provisions and
reconciles Talos clusters from YAML cluster templates. It runs in the `amiya.akn` core
platform cluster and manages downstream clusters such as `lungmen.akn` and `rhodes.akn`.

The procedure covers the full creation lifecycle: allocating a cluster identity, generating
the cluster template from the reference base, verifying bootstrap-manifest reachability,
applying machine classes, validating and applying the template, monitoring convergence, and
retrieving the kubeconfig.

> \[!IMPORTANT]
> **Omni is hosted on `amiya.akn`, the production core platform.** This procedure only
> *consumes* Omni to provision a downstream cluster; it does not modify `amiya.akn` itself.

### Technical framework and conventions

Omni manages Talos clusters through three independent building blocks:

* **Cluster templates** (`*.clustertemplate.yaml`) — Declarative YAML describing the desired
  cluster: name, Kubernetes and Talos versions, control-plane and worker machine classes, pod
  and service CIDRs, and bootstrap manifest URLs. These are **flat multi-doc YAML** with only
  a top-level `kind` and `name` per document — there is **no `apiVersion`, `metadata`, or
  `spec`** field. This format is **incompatible with kustomize** (which requires `metadata`),
  so templates are **standalone full copies edited in place**, never kustomize overlays.
* **Machine classes** (COSI resources) — Define VM shapes (vCPU, memory, disk) and Proxmox
  placement (bridges, VLANs, tags). They are **independent of cluster templates** and applied
  separately via `omnictl apply`. They live in
  `projects/chezmoi.sh/src/infrastructure/omni/machineclasses/`.
* **Bootstrap manifests** — Referenced by commit SHA in the template through
  `raw.githubusercontent.com` URLs. Cilium (the CNI) and metric-server are applied at
  bootstrap. The pinned SHA **must be reachable on GitHub's `main` branch** at apply time, or
  the CNI is silently never installed (see [Known issues](#known-issues)).

`omnictl` is provisioned through `mise` (`mise install`). The `OMNICONFIG` environment
variable is set after authenticating with Omni's OIDC admin flow
(`mise run bao:login:admin` from `projects/chezmoi.sh/`).

### Pod CIDR allocation

ADR-014 defines the pod CIDR supernet (`172.30.0.0/16`) and the principle that each
cluster allocates a non-overlapping `/19` from it. The authoritative per-cluster
allocation table lives in [`docs/network/vlans.md`](../../network/vlans.md). The next
available CIDR is picked by incrementing the cluster ID:

| Cluster ID | Cluster  | Pod CIDR          |
| ---------- | -------- | ----------------- |
| 1          | rhodes   | `172.30.0.0/19`   |
| 2          | lungmen  | `172.30.32.0/19`  |
| 3          | *(next)* | `172.30.64.0/19`  |
| …          | …        | …                 |
| 8          | sandbox  | `172.30.224.0/19` |

The **service CIDR `172.31.0.0/19`** and **clusterDNS `172.31.0.10`** are **shared across all
clusters** (ClusterMesh-ready). Do not change them.

### Prerequisites

Before starting, ensure:

* **Proxmox VE SDN is operational** — the shared `talosnet` VNet, its subnet (gateway/SNAT/
  DHCP), and the `SDN.Use` ACL for `omni@pve` exist. See
  [INF-20260627-00](../infrastructure/INF-20260627-00.proxmox-sdn-setup.md).
* **`omnictl` available** (`mise install`).
* **Omni authentication** — `OMNICONFIG` set via `mise run bao:login:admin` from
  `projects/chezmoi.sh/`.
* **Machine classes applied in Omni** — they are COSI resources, not part of the template. If
  Omni was reset, re-apply them (Step 4).
* **The pinned commit SHA is reachable on `main`** — see Step 3.

## Step 1 — Allocate the cluster identity

Pick the next cluster ID and its corresponding pod CIDR from the
[allocation table](#pod-cidr-allocation). For example, with `lungmen` (ID 1)
and `rhodes` (ID 2) taken, the next cluster is ID 3 → pod CIDR `172.30.64.0/19`.

Choose a cluster name (e.g. `rhodes-akn`) and a project directory
(`projects/<cluster>.akn/`).

> \[!IMPORTANT]
> **Do not change the service CIDR or clusterDNS.** `serviceSubnets` must stay
> `172.31.0.0/19` and `clusterDNS` must stay `172.31.0.10`. These are shared across all
> clusters so that ClusterMesh can connect them. Only the pod CIDR (`podSubnets`) differs per
> cluster.

***

## Step 2 — Create the cluster template

Copy the reference base template into the cluster's project and override the fields in place:

```sh
cp catalog/omni/clustertemplates/base.yaml \
   projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml
```

Fields to override from the base:

* `kind: Cluster` → `name:` — the cluster name (e.g. `rhodes-akn`).
* `podSubnets:` — the cluster's allocated `/19` (e.g. `172.30.64.0/19`).
* `controlPlane.machineClass.size:` — number of control-plane nodes (usually `1` or `3`).
* `workers.machineClass.size:` — number of worker nodes.
* Verify `serviceSubnets` stays `172.31.0.0/19` and `clusterDNS` stays `172.31.0.10`.

> \[!IMPORTANT]
> **Templates cannot be kustomized.** Omni cluster templates are flat multi-doc YAML whose
> documents carry only a top-level `kind` and `name` — there is no `apiVersion`, `metadata`,
> or `spec`. Because kustomize requires `metadata`, these templates **cannot** be used as
> kustomize overlays. Each cluster gets a **full standalone copy** of the base template,
> edited in place. Do not attempt to share a template via `kustomization.yaml`.

***

## Step 3 — Verify SHA reachability

Before the template can bootstrap, the pinned commit SHA must exist on GitHub's `main`
branch. After any branch is squash-merged, confirm the URL resolves:

```sh
curl -sI "https://raw.githubusercontent.com/chezmoidotsh/arcane/<SHA>/catalog/talos/manifests/cilium/1.19.5-native.yaml" | head -1
# → HTTP/2 200
```

If the URL returns `404`, the SHA is not reachable (the branch tip was not preserved on
`main` after squash-merge). It must be repinned to the merge-commit SHA before proceeding —
see [OMNI-20260629-03 — SHA pinning](./OMNI-20260629-03.sha-repin.md).

> \[!WARNING]
> Applying a template whose pinned SHA 404s produces a cluster with **no CNI**. Cilium never
> downloads, pods stay in `ContainerCreating`, and the failure is silent. Always verify
> reachability here, before Step 6.

***

## Step 4 — Apply machine classes

Machine classes are COSI resources applied via Omni, independent of the cluster template.
They are shared infrastructure defined in the `chezmoi.sh` project:

```sh
cd projects/chezmoi.sh
mise run omni:machineclass:diff     # dry-run / preview
mise run omni:machineclass:apply    # apply all machine classes to Omni
```

Authentication is required first — `mise run bao:login:admin` sets `OMNICONFIG`.

> \[!NOTE]
> If the required machine classes already exist in Omni, this step can be skipped. Verify
> them with `mise run omni:machineclass:list`.

***

## Step 5 — Validate the cluster template

Run offline validation to confirm the template is well-formed:

```sh
omnictl cluster template validate -f projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml
```

In `projects/lungmen.akn/` there is a wrapper mise task:

```sh
mise run omni:clustertemplate:validate
```

> \[!NOTE]
> This mise wrapper exists only in `projects/lungmen.akn/`. For a new cluster
> project, use the raw `omnictl cluster template validate -f <path>` command
> directly, or add an equivalent task to the new project's `.mise.toml`.

***

## Step 6 — Apply the cluster template

Once validated, apply the template to Omni. This step is online and requires authentication:

```sh
cd projects/chezmoi.sh && mise run bao:login:admin   # authenticate (sets OMNICONFIG)
omnictl apply -f projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml
```

> \[!IMPORTANT]
> Use `omnictl apply -f`, **not** `omnictl cluster template sync`. Both the template file
> headers and `catalog/omni/README.md` specify `omnictl apply -f` as the application command.

***

## Step 7 — Monitor cluster convergence

Omni begins provisioning VMs on Proxmox and bootstrapping Talos. Monitor convergence:

```sh
omnictl cluster status <cluster-name>
omnictl machine status
```

Convergence typically takes 5–15 minutes depending on VM startup time. Watch for all nodes to
reach the `Ready` state and for the control plane to form quorum.

> \[!NOTE]
> If convergence stalls beyond 15 minutes, check the Omni UI for machine-level errors.
> Common causes: Proxmox resource exhaustion, unreachable Talos endpoints, or missing machine
> classes (re-apply via Step 4).

***

## Step 8 — Retrieve the kubeconfig and verify

Once the cluster shows `Ready`, download the kubeconfig and verify node and pod health:

```sh
omnictl kubeconfig <cluster-name> > ~/.kube/config-<cluster-name>
export KUBECONFIG=~/.kube/config-<cluster-name>

kubectl get nodes -o wide
kubectl get pods -A
```

Verify Cilium is running:

```sh
kubectl get pods -n kube-system | grep cilium
```

> \[!WARNING]
> **Known Cilium behaviour.** When Cilium starts in native (kube-proxy-replacement) mode,
> there is a brief (\~2 minute) window where node-to-node connectivity is interrupted while
> Cilium replaces kube-proxy. This is expected and self-resolves — do not intervene. If
> Cilium pods never appear at all, the pinned SHA was unreachable; see
> [Known issues](#known-issues) below.

***

## Known issues

### CNI auto-apply may silently fail

If the pinned SHA is unreachable (HTTP 404), Cilium never downloads and the cluster has no
networking. Pods stay in `ContainerCreating`. Mitigation: verify SHA reachability (Step 3)
before applying. If it has already failed, repin the SHA
([OMNI-20260629-03](./OMNI-20260629-03.sha-repin.md)) and re-apply the template.

### \~2-minute connectivity blip during Cilium kubeProxyReplacement takeover

When Cilium starts in native (kube-proxy-replacement) mode, there is a brief window where
node-to-node connectivity is interrupted while Cilium replaces kube-proxy. This is expected
and self-resolves.

### Machine classes are erased on Omni reset

If the Omni instance is reset or reinstalled, all machine classes are lost. They must be
re-applied (`mise run omni:machineclass:apply` from `projects/chezmoi.sh/`) before any
cluster template can converge. The cluster templates themselves persist in Git.

### SHA pinning caveat

Always ensure the pinned SHA is reachable on `main` after a squash-merge. A branch-only SHA
will 404 once the branch commits are garbage-collected. See
[OMNI-20260629-03 — SHA pinning](./OMNI-20260629-03.sha-repin.md).
