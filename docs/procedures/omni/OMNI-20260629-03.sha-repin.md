## Procedure OMNI-20260629-03: SHA pinning ‚Äî repinning Omni bootstrap manifests after a squash-merge

This procedure covers the SHA-pinning strategy used in Omni cluster templates and the steps to repin Talos bootstrap
manifest references after a pull request is squash-merged to `main`.

Omni cluster templates reference the Talos bootstrap manifests (Cilium, metric-server) by exact Git commit SHA through
`raw.githubusercontent.com` URLs. Omni's bootstrap process fetches these URLs at cluster creation and reconciliation
time. If the SHA is unreachable ‚Äî the URL returns HTTP 404 ‚Äî the manifest never downloads, and critical components like
Cilium (the CNI) are silently never installed. The cluster then appears stuck, waiting indefinitely for networking that
will never arrive.

### Why SHA pinning exists

- **Reproducibility** ‚Äî Every cluster bootstraps from the exact same manifest bytes, so two clusters created weeks apart
  are identical at the CNI layer.
- **Security** ‚Äî An immutable commit reference cannot be silently changed; altering a manifest always requires an
  explicit template update.
- **Auditability** ‚Äî The exact version of every applied manifest is visible directly in the template, next to the URL.

### The squash-merge problem

The trade-off of SHA pinning surfaces on squash-merge. GitHub's squash-merge creates a **new** commit on `main` with a
SHA different from the branch tip. The original branch commits ‚Äî including the one the template was pinned to ‚Äî become
unreferenced and are eventually garbage-collected by GitHub. When that happens,
`raw.githubusercontent.com/<org>/<repo>/<sha>/...` returns HTTP 404. The sequence is:

1. The original commit SHA (e.g. `06e860db‚Ä¶`) no longer exists in `main`'s history.
2. The squash-merge creates a commit with a different SHA.
3. Talos nodes that fetch `raw.githubusercontent.com/.../06e860db‚Ä¶/...` receive a 404.
4. Cilium is never installed, and the cluster has no networking.

### Technical framework and conventions

- **SHA-pinned URLs** ‚Äî Bootstrap manifest URLs embed a 40-character commit SHA:
  `https://raw.githubusercontent.com/chezmoidotsh/arcane/<SHA>/catalog/talos/manifests/...`. The SHA must exist on
  GitHub's `main` branch for the URL to resolve.
- **Squash-merge behaviour** ‚Äî A squash-merge produces one new commit on `main`. The original branch commits are not
  part of `main`'s history and are garbage-collected once no references remain.
- **Tools** ‚Äî `git` (read the merge commit), `curl` (verify URL reachability), `sed` (rewrite the SHA), and `omnictl`
  (validate and apply the template). `omnictl` is provisioned through `mise`.

### Prerequisites

Before starting, ensure:

- **The pull request has been squash-merged to `main`** and the merge commit is pushed to `origin/main`.
- **Local `git` access** to the `chezmoidotsh/arcane` repository, with `main` fetched (`git fetch origin`).
- **`curl`** available to verify manifest reachability.
- **`omnictl`** available (`mise install`) for template validation and application.
- **Omni authentication** ‚Äî `OMNICONFIG` set via `mise run bao:login:admin` from `projects/chezmoi.sh/` (required for
  Step 5 only).

### Files involved

- **Cluster templates**: `projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml` (one per cluster; e.g.
  `projects/lungmen.akn/src/infrastructure/omni/lungmen.clustertemplate.yaml`).
- **Reference base template**: `catalog/omni/clustertemplates/base.yaml`.
- **Bootstrap manifests referenced by SHA**: `catalog/talos/manifests/` ‚Äî e.g. `cilium/1.19.5-native.yaml` and
  `metric-server/0.8.1.yaml`.

## Step 1 ‚Äî Identify the merge commit on `main`

After the squash-merge, obtain the commit SHA that now lives on `main`. This is the SHA that will persist and remain
reachable on GitHub:

```sh
git fetch origin
git log --oneline -1 origin/main
# e.g.  a1b2c3d4e5f6...  *[scope]: Subject
```

Record the **full 40-character SHA** of the merge commit shown on `origin/main`. This is the value the template must be
repinned to.

> \[!NOTE] Read from `origin/main` (the remote) rather than your local `main` to guarantee you are reading the commit
> GitHub actually serves through `raw.githubusercontent.com`.

---

## Step 2 ‚Äî Verify reachability

Before touching the template, confirm each pinned URL resolves with the new SHA. Both must return `HTTP/2 200`:

```sh
curl -sI "https://raw.githubusercontent.com/chezmoidotsh/arcane/<NEW_SHA>/catalog/talos/manifests/cilium/1.19.5-native.yaml" | head -1
curl -sI "https://raw.githubusercontent.com/chezmoidotsh/arcane/<NEW_SHA>/catalog/talos/manifests/metric-server/0.8.1.yaml" | head -1
```

> \[!IMPORTANT] **Do not continue if either URL returns 404.** A 404 means the SHA is not yet reachable ‚Äî either the
> merge commit has not been pushed to `main`, or GitHub's CDN has not refreshed yet (typically resolves within a minute
> of the push). Re-run `git fetch origin`, confirm the commit is on `origin/main`, and retry. Repinning to an
> unreachable SHA reproduces the exact failure this procedure exists to fix.

---

## Step 3 ‚Äî Update the template

Replace the old SHA with the new one in every cluster template file. There may be several ‚Äî one per cluster under
`projects/<cluster>/src/infrastructure/omni/`, plus the reference base in `catalog/omni/clustertemplates/base.yaml`:

```sh
OLD_SHA="06e860db4df0adfc02133bc5f87ab1e541bf5b4b"
NEW_SHA="<merge-commit-sha-from-step-1>"

# Per-cluster templates
sed -i "s/${OLD_SHA}/${NEW_SHA}/g" \
  projects/*/src/infrastructure/omni/*.clustertemplate.yaml

# Reference base template (if it also carries the pinned SHA)
sed -i "s/${OLD_SHA}/${NEW_SHA}/g" \
  catalog/omni/clustertemplates/base.yaml
```

> \[!NOTE] **macOS / BSD `sed`.** The `-i` flag on BSD `sed` (the macOS default) requires a backup suffix argument: use
> `sed -i ''` (empty string) instead of `sed -i`. GNU `sed` (Linux, Nix) takes bare `sed -i`.

---

## Step 4 ‚Äî Validate the template

Run offline validation to confirm the template is well-formed after the edit:

```sh
omnictl cluster template validate -f projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml
```

In `projects/lungmen.akn/` there is a wrapper mise task:

```sh
mise run omni:clustertemplate:validate
```

> \[!NOTE] This mise wrapper exists only in `projects/lungmen.akn/`. For a new cluster project, use the raw
> `omnictl cluster template validate -f <path>` command directly, or add an equivalent task to the new project's
> `.mise.toml`.

> \[!NOTE] Validation catches malformed YAML and invalid manifest paths, but it does **not** verify that the pinned SHA
> is reachable on GitHub ‚Äî that is what Step 2 is for. Always perform the Step 2 reachability checks before applying.

---

## Step 5 ‚Äî Apply the updated template

Once validated, apply the repinned template to Omni. This step is online and requires Omni authentication:

```sh
# Authenticate first (sets OMNICONFIG)
cd projects/chezmoi.sh && mise run bao:login:admin

# Apply the repinned template
omnictl apply -f projects/<cluster>/src/infrastructure/omni/<name>.clustertemplate.yaml
```

> \[!IMPORTANT] Use `omnictl apply -f`, **not** `omnictl cluster template sync`. Both the template file headers and
> `catalog/omni/README.md` specify `omnictl apply -f` as the application command.

## Recommended CI guard

Add a CI job that fails before a broken SHA can land on `main`. The guard extracts every 40-character hex SHA from
`*.clustertemplate.yaml` files and verifies each one exists in the local clone:

```sh
# Extract all pinned SHAs from cluster templates and verify each is reachable
grep -ohE '[0-9a-f]{40}' projects/*/src/infrastructure/omni/*.clustertemplate.yaml \
  | sort -u \
  | while read -r sha; do
      if ! git cat-file -t "$sha" >/dev/null 2>&1; then
        echo "BROKEN SHA: $sha is not reachable in this clone" >&2
        exit 1
      fi
    done
```

How it works:

1. **Extract** all 40-character hex SHAs from every `*.clustertemplate.yaml` (regex `[0-9a-f]{40}`).
2. **Verify** each with `git cat-file -t <SHA>` ‚Äî the commit object must exist in the local clone.
3. **Fail** the CI run if any SHA is unreachable (a branch-only SHA that will 404 after squash-merge).

> \[!TIP] Scope the job to PRs that touch `*.clustertemplate.yaml` (path filter) and run it against the _merged_ result
> ‚Äî GitHub's `pull_request` merge commit or a `push` to `main` ‚Äî so it checks the SHA that will actually live on `main`.

## Quick verifications

- **SHA in the template**: `grep -ohE '[0-9a-f]{40}' projects/*/src/infrastructure/omni/*.clustertemplate.yaml` ‚Äî every
  SHA must resolve to a commit on `main`.
- **Reachability**:
  `curl -sI "https://raw.githubusercontent.com/chezmoidotsh/arcane/<SHA>/catalog/talos/manifests/cilium/1.19.5-native.yaml" | head -1`
  ‚Äî must return `HTTP/2 200`.
- **Template applied**: `omnictl get cluster <name>` ‚Äî the cluster reconciles against the repinned manifests.
