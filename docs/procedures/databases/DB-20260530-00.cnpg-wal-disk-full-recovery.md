## Procedure DB-20260530-00: CNPG WAL disk full recovery

This procedure covers the recovery of a CloudNative-PG cluster whose WAL PVC has reached 100% capacity, causing CNPG to
refuse starting postgres (`Not enough disk space`). It addresses both the immediate emergency (expanding the PVC to
unblock postgres) and the root cause most commonly responsible for silent WAL accumulation: the `Expected empty archive`
error from `barman-cloud-check-wal-archive`.

This procedure is safe to run on a live cluster. The PVC expansion is a live Longhorn online resize (no pod restart
required). The S3 move operation does not interrupt any running postgres process.

**Origin:** This procedure was derived from incident `docs/incidents/2026-05-30-cnpg-wal-disk-full-apps-secured.md`.

### Technical framework and conventions

All commands use `kubectl` with an explicit `--context` flag. Never rely on the current kubeconfig default context —
CNPG operations on the wrong cluster are hard to reverse.

The `mc` (MinIO Client) CLI is used for S3 operations. It is installed via `mise`:

```sh
mise install mc
mise x mc -- mc --version  # verify
```

barman-cloud logs are JSON-structured. Filtering with `python3` is preferred over `grep` to avoid matching unrelated log
lines.

`kubectl get backup` is ambiguous: both Longhorn and CNPG expose a `backup` resource. Always qualify:
`kubectl get backup.postgresql.cnpg.io`.

### Prerequisites

Before starting, ensure:

- `kubectl` is accessible with a context pointing to the affected cluster.
- `mise` environment is active (`mise install` has been run).
- You have read access to the `databases` namespace (or whichever namespace the cluster is in).
- You have write access to the S3 bucket (credentials are in the `cnpg-backup-credentials` secret in the cluster
  namespace).

### Required inputs

- `CLUSTER_CONTEXT`: kubectl context for the cluster (e.g., `admin@lungmen.akn`).
- `CLUSTER_NAME`: CNPG cluster name (e.g., `apps-secured-20260329`).
- `NAMESPACE`: Kubernetes namespace (e.g., `databases`).

## Step 1 — Diagnose cluster state

Identify the cluster phase, conditions, and pod states to confirm this procedure applies.

```sh
# Check cluster phase and conditions
kubectl --context <CLUSTER_CONTEXT> get cluster <CLUSTER_NAME> -n <NAMESPACE>
# → phase: "Not enough disk space"  ← this procedure applies

# Get full conditions for root cause hints
kubectl --context <CLUSTER_CONTEXT> get cluster <CLUSTER_NAME> -n <NAMESPACE> \
  -o jsonpath='{.status.conditions}' | python3 -m json.tool
# → look for ContinuousArchivingFailing with a message referencing "Expected empty archive"

# Check pod states
kubectl --context <CLUSTER_CONTEXT> get pods -n <NAMESPACE> | grep <CLUSTER_NAME>
# → pod-1 may be in CrashLoopBackOff; pod-2 (primary) may still be Running
```

Expected findings when this procedure applies:

- Cluster phase: `Not enough disk space`
- Condition: `ContinuousArchivingFailing` (type) with message containing `Expected empty archive` or
  `WAL archive check failed`
- At least one pod in `CrashLoopBackOff`

> \[!NOTE] **High restart count slows recovery.** With 500+ restarts, the CrashLoopBackOff backoff is \~5 minutes. After
> the PVC expansion in Step 2, the pod will not restart immediately — wait for the next backoff cycle.

> \[!WARNING] **If the condition shows a different error** (e.g., a storage driver issue, a missing secret, or a network
> error), this procedure does not apply. Stop here and investigate the actual condition message before proceeding.

## Step 2 — Check WAL PVC usage and expand

Confirm the WAL PVCs are full, then expand them to provide enough headroom for postgres to start and for archived WALs
to drain.

```sh
# Check PVC sizes and usage
kubectl --context <CLUSTER_CONTEXT> get pvc -n <NAMESPACE> | grep <CLUSTER_NAME>
# → look for <CLUSTER_NAME>-1-wal and <CLUSTER_NAME>-2-wal at 100%

# Expand both WAL PVCs (double the current size; adjust if needed)
kubectl --context <CLUSTER_CONTEXT> patch pvc <CLUSTER_NAME>-1-wal \
  -n <NAMESPACE> --type=json \
  -p '[{"op":"replace","path":"/spec/resources/requests/storage","value":"10Gi"}]'

kubectl --context <CLUSTER_CONTEXT> patch pvc <CLUSTER_NAME>-2-wal \
  -n <NAMESPACE> --type=json \
  -p '[{"op":"replace","path":"/spec/resources/requests/storage","value":"10Gi"}]'

# Verify expansion applied (Longhorn expands online — no restart needed)
kubectl --context <CLUSTER_CONTEXT> get pvc -n <NAMESPACE> | grep <CLUSTER_NAME>
```

> \[!NOTE] Longhorn supports online PVC resize: the filesystem is expanded at the next mount point access without a pod
> restart. The pod will pick up the new space after its next restart cycle.

> \[!IMPORTANT] **PVC expansion is irreversible.** You cannot shrink a PVC after expanding it. Choose a size that
> provides sufficient headroom (at minimum, enough to drain existing WALs plus 2–3 days of generation). For the
> `apps-secured` cluster, 10 Gi was sufficient.

Wait for pods to return to `Running` before proceeding. With a high restart count, expect up to 5 minutes.

```sh
# Wait for pods to be Running
kubectl --context <CLUSTER_CONTEXT> get pods -n <NAMESPACE> -w | grep <CLUSTER_NAME>
```

## Step 3 — Identify the root cause in barman-cloud logs

With postgres running again, check the barman-cloud sidecar logs to confirm the `Expected empty archive` error.

```sh
# Identify the primary pod (it has 2/2 containers Running)
kubectl --context <CLUSTER_CONTEXT> get pods -n <NAMESPACE> | grep <CLUSTER_NAME>
# note the pod-2 name (or whichever is the primary)

# Filter barman-cloud logs for errors (JSON log format — filter out "retention" noise)
kubectl --context <CLUSTER_CONTEXT> logs -n <NAMESPACE> <PRIMARY_POD> \
  -c plugin-barman-cloud --since=15m 2>/dev/null \
  | python3 -c "
import sys, json
for line in sys.stdin:
    line = line.strip()
    if not line: continue
    try:
        d = json.loads(line)
        msg = d.get('msg', '')
        if 'retention' not in msg.lower():
            print(f'[{d.get(\"level\",\"\")}] {d.get(\"ts\",\"\")[:19]}: {msg[:300]}')
    except:
        pass
" | tail -30
```

Expected output when this procedure applies:

```text
[error] 2026-05-30T17:25:00: WAL archive check failed for server 01KMWETKMBK0MYNB3X7RH6BT0X:
  Expected empty archive
```

> \[!NOTE] `barman-cloud-check-wal-archive` runs on **every pod restart**, not only at cluster creation. A failover that
> leaves data in the S3 `serverName/` path will permanently block archiving until the path is cleared.

If the error is **not** `Expected empty archive`, stop here. The S3 move in Step 4 does not apply to other error types.

## Step 4 — Identify the S3 path and prepare mc

Extract the S3 coordinates from the cluster's plugin configuration and ObjectStore.

```sh
# Get the serverName from the cluster spec
kubectl --context <CLUSTER_CONTEXT> get cluster <CLUSTER_NAME> -n <NAMESPACE> \
  -o jsonpath='{.spec.plugins[?(@.name=="barman-cloud.cloudnative-pg.io")].parameters.serverName}'
# → e.g. 01KMWETKMBK0MYNB3X7RH6BT0X

# Get the destinationPath from the ObjectStore (find the ObjectStore name first)
kubectl --context <CLUSTER_CONTEXT> get objectstore -n <NAMESPACE>
kubectl --context <CLUSTER_CONTEXT> get objectstore <OBJECTSTORE_NAME> -n <NAMESPACE> \
  -o jsonpath='{.spec.configuration.destinationPath}'
# → e.g. s3://cnpg-backups/lungmen.akn/apps-secured

# Extract S3 credentials from the secret
kubectl --context <CLUSTER_CONTEXT> get secret cnpg-backup-credentials \
  -n <NAMESPACE> -o json \
  | python3 -c "
import sys, json, base64
s = json.load(sys.stdin)
for k, v in s['data'].items():
    print(f'{k}={base64.b64decode(v).decode()}')
"
# → access_key_id=...
# → access_secret_key=...
# → endpoint_url=https://s3.chezmoi.sh:9000
# → region=us-east-1

# Install mc if not available
mise install mc

# Configure mc alias
mise x mc -- mc alias set myminio <endpoint_url> <access_key_id> <access_secret_key> --api S3v4

# Verify: list the destination path to see the serverName folder
mise x mc -- mc ls myminio/<bucket>/<path>/
# → should show the serverName directory with objects inside
```

## Step 5 — Move the stale S3 folder to .bak

Move the serverName directory to a `.bak` suffix instead of deleting it. This preserves the original base backup as a
safety net until the new cluster is confirmed stable.

```sh
SERVER_NAME="<serverName from Step 4>"
BUCKET_PATH="<bucket/path from Step 4>"

# Move to .bak (preserves data, clears the path for the check)
mise x mc -- mc mv --recursive \
  myminio/${BUCKET_PATH}/${SERVER_NAME}/ \
  myminio/${BUCKET_PATH}/${SERVER_NAME}.bak/

# Verify: the original path should now be empty or absent
mise x mc -- mc ls myminio/${BUCKET_PATH}/
# → only <SERVER_NAME>.bak/ should appear momentarily
```

> \[!WARNING] Do not `mc rm` the folder. The `.bak` copy protects against data loss if the new base backup fails before
> completing. Delete `.bak` only after a new base backup has been verified completed.

## Step 6 — Verify WAL archiving is active

Wait for the next barman-cloud archive attempt and confirm it succeeds.

```sh
# Watch barman-cloud logs on the primary pod
kubectl --context <CLUSTER_CONTEXT> logs -n <NAMESPACE> <PRIMARY_POD> \
  -c plugin-barman-cloud -f 2>/dev/null \
  | python3 -c "
import sys, json
for line in sys.stdin:
    line = line.strip()
    if not line: continue
    try:
        d = json.loads(line)
        msg = d.get('msg', '')
        if 'retention' not in msg.lower():
            print(f'[{d.get(\"level\",\"\")}] {d.get(\"ts\",\"\")[:19]}: {msg[:200]}')
    except:
        pass
"
# → expect: [info] ...: Archived WAL file 000000020000000000000001
#            [info] ...: Archived WAL file 000000020000000000000002

# Confirm cluster condition has cleared
kubectl --context <CLUSTER_CONTEXT> get cluster <CLUSTER_NAME> -n <NAMESPACE> \
  -o jsonpath='{.status.conditions}' | python3 -m json.tool | grep -A3 -i archiv
# → ContinuousArchivingSuccess
```

> \[!NOTE] The replica pod (pod-1) may remain in `1/2 Running` for an extended period while it replays the backlog of
> unarchived WALs via streaming recovery. This is normal. The primary (pod-2) archiving successfully is sufficient to
> confirm the fix.

## Step 7 — Trigger an immediate backup

With archiving restored, create an on-demand backup to establish a clean recovery point before any migration or cluster
rebuild.

```sh
# Trigger backup via the cnpg-backup skill script
# Note: validate-cnpg.sh may report unhealthy if the replica is still replaying WALs.
# The primary is healthy — it is safe to skip validation and trigger directly.
backup_name="$(.agents/skills/cnpg-backup/scripts/backup-cnpg.sh \
  <CLUSTER_NAME> <NAMESPACE> <CLUSTER_CONTEXT>)"
echo "Created: $backup_name"

# Verify backup completion (use the full API group to avoid Longhorn ambiguity)
sleep 60
.agents/skills/cnpg-backup/scripts/verify-cnpg.sh \
  "$backup_name" <NAMESPACE> <CLUSTER_CONTEXT>
# → COMPLETED: <backup_name>

# Alternatively, check directly:
kubectl --context <CLUSTER_CONTEXT> get backup.postgresql.cnpg.io \
  "$backup_name" -n <NAMESPACE>
```

> \[!NOTE] Always use `kubectl get backup.postgresql.cnpg.io` — not `kubectl get backup`. Both Longhorn and CNPG expose
> a resource named `backup`. Without the API group qualifier, the wrong resource type is returned.

## Quick verifications

- **Cluster phase**: `kubectl --context <CTX> get cluster <NAME> -n <NS>` → `Cluster in healthy state`
- **WAL archiving**: cluster condition `ContinuousArchiving: True (ContinuousArchivingSuccess)`
- **Pods Running**: `kubectl --context <CTX> get pods -n <NS> | grep <NAME>` → both `2/2 Running`
- **Backup completed**: `kubectl --context <CTX> get backup.postgresql.cnpg.io <backup_name> -n <NS>` → `completed`
- **S3 new WALs**: `mise x mc -- mc ls myminio/<bucket>/<path>/<SERVER_NAME>/wals/` → new WAL files present

## Rollback

**PVC expansion** cannot be reversed — Longhorn does not support shrinking PVCs. This is acceptable: more WAL space does
not break anything.

**S3 move** can be reversed if the `.bak` folder exists:

```sh
# Restore the original serverName folder from .bak
mise x mc -- mc mv --recursive \
  myminio/${BUCKET_PATH}/${SERVER_NAME}.bak/ \
  myminio/${BUCKET_PATH}/${SERVER_NAME}/
```

Only revert the S3 move if you need to abandon this recovery path (e.g., switching to a full cluster rebuild from a
different backup source). Reverting will re-block WAL archiving.

---

## Long-term fix: architectural change

The S3 move resolves the immediate failure but does not prevent recurrence after the next failover. For homelab clusters
where WAL-based point-in-time recovery is not required, consider:

- **`isWALArchiver: false`** — disables WAL streaming to S3 entirely. WALs are recycled after checkpoints. Base backups
  (via `ScheduledBackup`) remain functional and are sufficient for daily backup coverage.
- **WAL PVC size 2 Gi** — adequate when WALs are not archived (recycled after checkpoint + `max_slot_wal_keep_size`).
- **`max_slot_wal_keep_size: 1500MB`** — prevents replica slot lag from filling a 2 Gi WAL PVC; invalidates the slot and
  triggers a `pg_basebackup` instead of crashing the primary.
- **Daily `ScheduledBackup`** with `immediate: true` — provides a daily recovery point without WAL streaming.

See `docs/incidents/2026-05-30-cnpg-wal-disk-full-apps-secured.md` for the full rationale.
