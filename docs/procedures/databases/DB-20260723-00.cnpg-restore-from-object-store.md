## Procedure DB-20260723-00: Restore a CNPG cluster from its S3 object-store backup

This procedure bootstraps a CloudNative-PG `Cluster` from an existing `barman-cloud.cloudnative-pg.io` plugin backup,
either to recreate the same logical database on a brand-new Kubernetes cluster (disaster recovery) or to roll a cluster
back to its last known-good backup. It is deliberately generic and app-agnostic — the
[OpenBao](../../../projects/rhodes.akn/docs/disaster-recovery/openbao.md) and
[Pocket-Id](../../../projects/rhodes.akn/docs/disaster-recovery/pocket-id.md) disaster recovery procedures both use it
as their database-restore step rather than duplicating the mechanics.

This is **not** the same operation as `.agents/skills/cnpg-backup/SKILL.md` (on-demand backup of a healthy, currently
running cluster) — that skill explicitly routes restore requests here instead of improvising them.

### Technical framework and conventions

Every CNPG cluster in this repo backed by the `barman-cloud.cloudnative-pg.io` plugin has three coordinates that this
procedure ties together, and they are easy to conflate:

- **`serverName`** (`spec.plugins[].parameters.serverName`, also under `externalClusters[].plugin.parameters`): a ULID
  identifying one specific S3 folder under the `ObjectStore`'s `destinationPath` — e.g.
  `s3://cnpg-rhodes-akn/openbao/01KEPSN912SSFFARX3YYZNA22Q/`. ULIDs are lexicographically sortable by creation time, so
  the **last** folder in an alphabetically sorted listing is the most recent.
- **`spec.bootstrap.recovery.source`**: an arbitrary label (this repo's convention is the restore's timestamp, e.g.
  `2026-01-11T08:07:53.126Z`) that must exactly match the `name` of one entry in `spec.externalClusters[]`.
- **`spec.externalClusters[].name`**: the label referenced by `bootstrap.recovery.source`. Its own
  `plugin.parameters.serverName` is what actually resolves to the S3 folder above.

In other words: `bootstrap.recovery.source` → looked up in `externalClusters[].name` → resolves via
`externalClusters[].plugin.parameters.serverName` → the real S3 folder. Three names, two of which you choose (the label)
and one of which you must read off S3 (the ULID).

> [!NOTE]
>
> This procedure uses `s3cmd` for S3 operations (as does
> [DB-20260530-00](DB-20260530-00.cnpg-wal-disk-full-recovery.md)) rather than the MinIO Client (`mc`) — `mc` requires a
> persistent `mc alias set` step, which is unnecessary overhead when this kind of procedure is typically run once,
> against a workstation with no prior Garage alias configured (a fresh DR drill, or a new operator machine). `s3cmd`
> takes credentials as flags on every invocation instead, with no setup step.

### Prerequisites

Before starting, ensure the following tools are installed and configured: `kubectl`, `s3cmd`, `kustomize`, `ksops` (for
decrypting the source app's SOPS secrets if not already applied).

You must also have:

- `kubectl` access to the target cluster with permission to create/patch resources in the target namespace.
- The CNPG operator and the `barman-cloud.cloudnative-pg.io` plugin already installed on the target cluster (out of
  scope here — part of the cluster's own bring-up, e.g.
  [OMNI-20260721-00](../omni/OMNI-20260721-00.omni-cluster-creation.md)).
- Read access to the S3 bucket backing the source `ObjectStore` (credentials come from the `cnpg-backup-credentials`
  secret in the app's namespace — apply the app's `sops/` source first if it isn't there yet).

### Required inputs

- `CLUSTER_CONTEXT`: kubectl context for the target cluster (e.g., `admin@rhodes.akn`).
- `NAMESPACE`: namespace the CNPG cluster lives in (e.g., `vault`).
- `CLUSTER_MANIFEST`: path to the `Cluster` manifest to restore (e.g.,
  `projects/rhodes.akn/src/apps/vault/openbao.postgresql.yaml`).
- `OBJECTSTORE_MANIFEST`: path to the matching `ObjectStore` manifest (same directory, `*-objectstore.yaml`).
- `SCHEDULEDBACKUP_MANIFEST`: path to the matching `ScheduledBackup` manifest (same directory, `*-backup.yaml`) — used
  in Step 4 to resume the restored cluster's own backup cadence.
- `CLUSTER_NAME`: the `Cluster` resource's `metadata.name` — not known upfront if you're renaming it in Step 3
  (`kubectl get cluster` after Step 4 confirms the actual name in use for Quick verifications/Rollback below).

## Step 1 — Extract S3 coordinates and credentials

```sh
# Bucket + path from the ObjectStore manifest
grep destinationPath <OBJECTSTORE_MANIFEST>
# → e.g. destinationPath: s3://cnpg-rhodes-akn/openbao

# Endpoint from the same manifest
grep endpointURL <OBJECTSTORE_MANIFEST>
# → e.g. endpointURL: https://s3.chezmoi.sh

# Credentials — decode from the already-applied cnpg-backup-credentials secret
kubectl --context <CLUSTER_CONTEXT> get secret cnpg-backup-credentials \
  -n <NAMESPACE> -o json \
  | python3 -c "
import sys, json, base64
s = json.load(sys.stdin)
for k, v in s['data'].items():
    print(f'{k}={base64.b64decode(v).decode()}')
"
# → access_key_id=...
# → access_secret_key=...
# → region=...
```

> [!NOTE]
>
> Not every app in this repo sources `cnpg-backup-credentials` the same way — check which applies before assuming the
> SOPS path below. Per
> [`@chezmoi.sh/pulumi-garage-cnpg-backup`](../../../catalog/pulumi/components/garage-cnpg-backup/README.md), most
> clusters get it from Vault via an `ExternalSecret` (in which case it exists once ESO is deployed and synced — no
> action needed here beyond confirming ESO is healthy). Only the core-platform cluster (which can't depend on Vault for
> its own backup credentials — chicken-and-egg) sources it from a SOPS-committed secret instead:
>
> ```sh
> kustomize build --enable-alpha-plugins --enable-exec <app_dir>/sops | kubectl --context <CLUSTER_CONTEXT> apply -f -
> ```
>
> Both the [OpenBao](../../../projects/rhodes.akn/docs/disaster-recovery/openbao.md) and
> [Pocket-Id](../../../projects/rhodes.akn/docs/disaster-recovery/pocket-id.md) disaster recovery procedures are on
> `rhodes.akn` (a core-platform cluster) and do this as their own Step 1 before calling into this procedure.

## Step 2 — List the bucket and identify the latest backup

> [!TIP]
>
> `mise run cnpg:backup:latest -- <context> <namespace> <objectstore_manifest> [credentials_secret]`
> (`.mise/tasks/cnpg/backup/latest`, repo-wide — works for any CNPG cluster backed by the
> `barman-cloud.cloudnative-pg.io` plugin, not just `rhodes.akn`) wraps this step and Step 1's credential extraction
> into one command. Run `mise run cnpg:backup:latest --help` for argument details. Read-only, same as this step. The
> `rhodes.akn` OpenBao/Pocket-Id disaster recovery docs wrap it further into per-app `dr:openbao:backup:latest` /
> `dr:pocket-id:backup:latest` tasks that hardcode the manifest path.

```sh
ACCESS_KEY="<access_key_id from Step 1>"
SECRET_KEY="<access_secret_key from Step 1>"
ENDPOINT="s3.chezmoi.sh"          # host only, no scheme
BUCKET_PATH="<bucket/path from Step 1, e.g. cnpg-rhodes-akn/openbao>"

# --host-bucket set equal to --host (no "%(bucket)s" template) forces path-style
# addressing rather than virtual-hosted — needed for Garage. Verify on first use;
# if listing fails, this is the first flag to double-check against the endpoint.
s3cmd --access_key="$ACCESS_KEY" --secret_key="$SECRET_KEY" \
  --host="$ENDPOINT" --host-bucket="$ENDPOINT" \
  ls "s3://${BUCKET_PATH}/"
# → one directory per serverName (ULID), e.g.:
#     DIR  s3://cnpg-rhodes-akn/openbao/01KEP1HV1693BFHSVVNCTQWZ7A/
#     DIR  s3://cnpg-rhodes-akn/openbao/01KEPSN912SSFFARX3YYZNA22Q/
# ULIDs sort lexicographically by creation time — the last one listed is the latest.

SERVER_NAME="<last/most-recent ULID from the listing>"

# Confirm it has both a base backup and WALs
s3cmd --access_key="$ACCESS_KEY" --secret_key="$SECRET_KEY" \
  --host="$ENDPOINT" --host-bucket="$ENDPOINT" \
  ls "s3://${BUCKET_PATH}/${SERVER_NAME}/"
# → expect "base/" and "wals/" subdirectories
```

## Step 3 — Rename the cluster and point it at that backup

This step touches four fields, and none of them are arbitrary — each follows a convention verified against the real
manifests in this repo:

- **`metadata.name`** gets a fresh `-YYYYMMDD` suffix (today's UTC date), replacing any existing one. CNPG
  restore-from-backup creates a **new** `Cluster` object rather than mutating the old one in place, so the new
  incarnation needs its own name — this is the pattern already visible in e.g. `pocket-id-20260530`.
- **`spec.bootstrap.recovery.source`** and **`spec.externalClusters[].name`** are **not** "the time this restore was
  run" — they are the decoded creation timestamp embedded in the recovery `serverName`'s own ULID (every existing
  occurrence in this repo's CNPG manifests matches its `serverName`'s decoded ULID timestamp exactly — ULIDs encode a
  48-bit millisecond timestamp in their first 10 Crockford-Base32 characters).
- **`spec.externalClusters[].plugin.parameters.serverName`** is the `<SERVER_NAME>` found in Step 2, with a trailing
  `# <decoded timestamp>` comment — the same decoded value as above.
- **`spec.plugins[].parameters.serverName`** (the cluster's own ongoing WAL archiver going forward) gets a **freshly
  generated** ULID, never the recovered one — see the `[!IMPORTANT]` callout below. Its own trailing comment is _its
  own_ decoded timestamp (effectively "now").

```yaml
metadata:
  name: <app>-<YYYYMMDD> # To adapt — replaces any existing "-YYYYMMDD" suffix
spec:
  bootstrap:
    recovery:
      source: "2026-01-11T08:07:53.126Z" # To adapt — decoded from SERVER_NAME's ULID, must match externalClusters[].name below
  plugins:
    - enabled: true
      isWALArchiver: true
      name: barman-cloud.cloudnative-pg.io
      parameters:
        barmanObjectName: <ObjectStore name, e.g. selfhosted>
        serverName: "<a FRESH ULID, not SERVER_NAME>" # To adapt — trailing comment = this ULID's own decoded timestamp
  externalClusters:
    - name: "2026-01-11T08:07:53.126Z" # Must match bootstrap.recovery.source exactly
      plugin:
        enabled: true
        isWALArchiver: false
        name: barman-cloud.cloudnative-pg.io
        parameters:
          barmanObjectName: <ObjectStore name, e.g. selfhosted>
          serverName: "<SERVER_NAME from Step 2>" # trailing comment = SERVER_NAME's decoded timestamp
```

> [!IMPORTANT]
>
> `isWALArchiver: false` on the `externalClusters[]` entry is deliberate — the cluster's own top-level `spec.plugins[]`
> entry (with the **new**, freshly-generated `serverName` above) is what archives WALs going forward. Do not reuse the
> recovered `serverName` there — it identifies a different, already-closed archive stream.

> [!TIP]
>
> For the `rhodes.akn` OpenBao/Pocket-Id call sites,
> `projects/rhodes.akn/docs/disaster-recovery/.mise/tasks/dr/cnpg/patch-recovery-source` (task
> `dr:cnpg:patch-recovery-source`, wrapped further by `dr:openbao:patch-recovery` / `dr:pocket-id:patch-recovery`)
> automates this entire step, including the ULID decode/generate logic above. It edits the file only — it does not
> `kubectl apply` anything. This task is local to that folder, not repo-wide like `cnpg:backup:latest` — there is no
> generic equivalent yet outside `rhodes.akn`.

> [!WARNING]
>
> Renaming `metadata.name` is **not self-contained to this file**. The matching `ScheduledBackup` manifest's
> `spec.cluster.name` (and any `app.kubernetes.io/instance` label elsewhere keyed on the old name) must be updated to
> match by hand — this script/step does not do it for you.

## Step 4 — Apply the ObjectStore and Cluster

```sh
kubectl --context <CLUSTER_CONTEXT> apply -f <OBJECTSTORE_MANIFEST>
kubectl --context <CLUSTER_CONTEXT> apply -f <CLUSTER_MANIFEST>

# Watch the restore proceed
kubectl --context <CLUSTER_CONTEXT> get cluster -n <NAMESPACE> -w
# → phase progresses: "Setting up primary" → "Recovering instance" → "Cluster in healthy state"
```

> [!NOTE]
>
> A restore can take anywhere from under a minute to tens of minutes depending on the base backup size and how much WAL
> has to replay since it was taken. Watch, don't assume it's stuck.

## Step 5 — Confirm the restored data

```sh
# Cluster phase
kubectl --context <CLUSTER_CONTEXT> get cluster -n <NAMESPACE>
# → Cluster in healthy state

# Pods
kubectl --context <CLUSTER_CONTEXT> get pods -n <NAMESPACE>
# → 2/2 Running (or 1/2 if a single-instance cluster, e.g. openbao-database)
```

Also apply the matching `ScheduledBackup` manifest at this point (same directory) so the restored cluster resumes its
own backup cadence instead of relying solely on the recovered history:

```sh
kubectl --context <CLUSTER_CONTEXT> apply -f <SCHEDULEDBACKUP_MANIFEST>
```

## Quick verifications

- **Cluster phase**: `kubectl --context <CTX> get cluster <CLUSTER_NAME> -n <NS>` → `Cluster in healthy state`
- **Pods Running**: `kubectl --context <CTX> get pods -n <NS>` → all containers `Running`
- **WAL archiving resumed**:
  `kubectl --context <CTX> get cluster <CLUSTER_NAME> -n <NS> -o jsonpath='{.status.conditions}'` →
  `ContinuousArchivingSuccess`
- **ScheduledBackup active**: `kubectl --context <CTX> get scheduledbackup -n <NS>`

## Rollback

If the wrong `serverName` was picked (e.g. an older backup, or a corrupt one) and the restore fails or comes up with
unexpected data:

```sh
# Delete the failed Cluster (and its PVCs — nothing on S3 is touched)
kubectl --context <CLUSTER_CONTEXT> delete cluster <CLUSTER_NAME> -n <NAMESPACE>
kubectl --context <CLUSTER_CONTEXT> delete pvc -n <NAMESPACE> -l cnpg.io/cluster=<CLUSTER_NAME>

# Repeat from Step 3 with the correct serverName
```

The S3 backup itself is never modified by this procedure — only read — so retrying is always safe.
