# Post-Mortem: Kubernetes 1.35 upgrade failure — removed feature gate UserNamespacesPodSecurityStandards causes kube-apiserver and kubelet crash on lungmen.akn

## Executive Summary

During the Kubernetes 1.34.0 → 1.35.0 upgrade on `lungmen.akn` (procedure `INF-20260525-01`), the kube-apiserver v1.35.0
crashed immediately at startup with `unrecognized feature gate: UserNamespacesPodSecurityStandards`. This feature gate
was removed in Kubernetes 1.35 but was still present in the cluster's Talos machine config patch. The kubelet v1.35.0
panicked for the same reason. A first remediation attempt using `talosctl patch machineconfig` failed because that
command applies a **strategic merge** — it adds or modifies keys but never deletes an existing key from a map. The
actual fix required extracting the full machine config, stripping the removed gate with a regex substitution, and
re-applying the config via `talosctl apply-config` to overwrite the node's state. No data loss occurred. The cluster was
fully operational at v1.35.0 within 45 minutes.

---

## Event Summary

**Expected outcome:** `talosctl upgrade-k8s --to 1.35.0` completes all phases without error, promoting all control plane
and data plane components to Kubernetes 1.35.0.

**Actual outcome:** kube-apiserver v1.35.0 exited immediately with
`unrecognized feature gate: UserNamespacesPodSecurityStandards`, taking the API server offline. The kubelet subsequently
panicked for the same reason when its own binary was upgraded. `kubectl` returned `connection refused` on port 6443 for
the entire remediation window.

**Impact:** kube-apiserver unavailable for \~45 min. No user-facing workload was affected because the upgrade was a
planned maintenance operation.

**Duration:** kube-apiserver down from \~09:30 UTC to \~10:16 UTC on 2026-05-26.

**First signal:** `talosctl containers -k` showing `CONTAINER_EXITED` for kube-apiserver immediately after the upgrade
step; `talosctl logs -k` revealing `unrecognized feature gate: UserNamespacesPodSecurityStandards`.

## What Went Well

- **Talos tooling stays operational independently of the kube-apiserver.** `talosctl containers -k`, `talosctl logs -k`,
  and `talosctl apply-config` all worked while port 6443 was returning `connection refused`. Diagnosis and remediation
  were entirely possible without `kubectl`.
- **`talosctl logs -k` surfaced the exact error immediately.** A single command revealed the unrecognized feature gate
  and eliminated any ambiguity about the root cause within minutes of the crash.
- **`talosctl upgrade-k8s` is idempotent.** After the config was corrected, the upgrade command could be relaunched
  without risk of double-applying or partial-state issues.
- **`kubelet-serving-cert-approver` self-healed.** That controller had been in `CrashLoopBackOff` for \~15 h before the
  upgrade; the restart cycle during the upgrade allowed pending CSRs to be approved, clearing the pre-existing issue
  without extra intervention.

## Root-Cause Analysis

**Technique:** 5 Whys **Why this technique:** Single linear cause chain — one missing pre-flight check that allowed an
unrecognized flag to reach the running binary.

### Analysis

1. **Why did kube-apiserver and kubelet v1.35.0 crash?** → The feature gate `UserNamespacesPodSecurityStandards` was
   present in the Talos machine config patch. That gate was **removed in Kubernetes 1.35** (it had been valid from 1.29
   to 1.34); passing an unrecognized gate is a fatal error for both components.

2. **Why was the removed feature gate still in the config?** → It was added when user namespaces support was first
   enabled and was never cleaned up. The upgrade procedure (`INF-20260525-01`) includes a step titled "checking for
   removed Kubernetes component flags", but `talosctl upgrade-k8s` only inspects flags for components it manages
   natively — it does not scan `extraArgs`, `extraConfig`, or feature gate maps injected through the user's Talos patch.

3. **Why did the first fix (merge patch) not resolve the issue?** → `talosctl patch machineconfig` implements a
   **strategic merge patch**: it fuses the provided YAML into the current machine config by adding or updating keys.
   Removing a key from the patch file has no effect — the key remains in the running config. This is the correct
   semantic for incremental updates but is silently wrong when the intent is to delete a key.

4. **Why was this merge behaviour not known in advance?** → The distinction between merge-patch semantics and
   replacement semantics is not documented in the upgrade procedure or in comments within `patch-config.yaml`. The
   default mental model — "removing from the file removes it from the node" — is incorrect for this command but is not
   contradicted anywhere in the local documentation.

5. **Why was there no guard against this class of failure?** → No step in the pre-upgrade checklist validates the
   feature gates present in the Talos patch against the set of gates removed or graduated in the target Kubernetes
   version. The existing "removed flags" check is scoped to Talos-native component flags and produces no output for
   gates embedded in user patches.

### Root Causes

- **`UserNamespacesPodSecurityStandards` not removed despite its removal from Kubernetes 1.35** — no mechanism validates
  the compatibility of feature gates in the Talos patch against the target Kubernetes version. The existing pre-upgrade
  check covers Talos-native flags only.
- **`talosctl patch machineconfig` does not delete keys** — the merge-patch behaviour creates a silent gap between
  intent (delete a key) and effect (key remains). This behaviour is undocumented in the upgrade procedure, making it a
  recurring latent risk for any remediation that requires key removal.

### Contributing Factors

- The upgrade procedure `INF-20260525-01` focuses on Talos-managed component flags and does not cover `extraArgs`,
  `extraConfig`, or feature gate maps supplied via the user's machine config patch.
- JSON6902 patches, which would allow a clean `op: remove`, are not supported for multi-document Talos machine
  configurations, leaving `apply-config` with a full-config extraction and overwrite as the only reliable deletion path.
- `patch-config.yaml` contains no comments linking feature gates to the Kubernetes versions in which they were
  introduced, graduated, or removed.

## Control Analysis

### In Control (what we could have changed)

- **No feature-gate lifecycle tracking in `patch-config.yaml`.** Each gate was added when needed and never annotated
  with its introduction or removal version. A deliberate choice to keep the file terse; the cost materialized here.
- **Upgrade procedure scoped to Talos-native flags only.** The pre-upgrade "removed flags" check was designed for
  components that `talosctl upgrade-k8s` manages directly. The decision to scope it narrowly left user-supplied
  `extraArgs` and `feature-gates` unvalidated.
- **No cross-reference between Kubernetes changelogs and the patch file before upgrading.** A manual review step — even
  a 5-minute grep against the changelog — would have caught `UserNamespacesPodSecurityStandards` before the upgrade ran.

### Out of Control (external factors)

| External factor                                                                   | What would reduce exposure?                                                                                 |
| --------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| Kubernetes upstream removes feature gates at its own release cadence              | Track each gate's graduation/removal version in `patch-config.yaml`; review before every minor version bump |
| `talosctl upgrade-k8s` does not inspect user-supplied `extraArgs`/`feature-gates` | Maintain a local pre-upgrade checklist that explicitly covers user-patch content                            |

## Systemic Lessons

- **Pre-upgrade checks for removed feature gates must cover user-supplied config, not just Talos-native flags.** The
  existing step produces a clean bill of health for everything that `talosctl upgrade-k8s` manages, but is silent on
  feature gates injected through the machine config patch. Any gate whose name does not appear in Talos's own component
  manifests will pass the check undetected, regardless of whether it is valid for the target Kubernetes version. This
  review must be done manually by cross-referencing the patch content against the Kubernetes changelog for the target
  version.

- **`talosctl patch machineconfig` is an additive operation, not a declarative one.** Using it to "remove" a setting by
  omitting it from the patch file produces no error and no effect on the existing key. When the goal is to delete a key
  from the running machine config, the correct procedure is: extract the full config, edit it, and re-apply via
  `talosctl apply-config`. This distinction should be explicit in every runbook that touches machine config remediation.

- **Operating without the kube-apiserver is possible and worth practicing.** The entire diagnosis and fix in this
  incident was performed through `talosctl` alone. Familiarity with `talosctl containers -k`, `talosctl logs -k`, and
  `talosctl apply-config` is a prerequisite for handling control-plane failures confidently.

## Agent Knowledge

- `talosctl patch machineconfig`: applies a **strategic merge** — adds/updates keys but never deletes. To remove a key,
  extract full config (`talosctl get machineconfig -o yaml`), edit, then `talosctl apply-config --file <path>`.
- `talosctl upgrade-k8s`: idempotent. After fixing config, relaunch is safe.
- Talos control plane diagnostics work without kube-apiserver: `talosctl containers -k`, `talosctl logs -k`,
  `talosctl apply-config` all use the Talos API directly.
- Kubernetes feature gates in user-supplied Talos patches are **not** validated by `talosctl upgrade-k8s --dry-run`. The
  "removed component flags" check covers Talos-native flags only.
- JSON6902 patches with `op: remove` are **not supported** for multi-document Talos machine configs.
