# Post-Mortem: Cilium 1.19 upgrade failure — amiya.akn node unreachable

## Executive Summary

A planned upgrade of Cilium from 1.18.10 to 1.19.4 on the `amiya.akn` single-node cluster failed across 8 attempts over
3 days. Each attempt caused total node connectivity loss within 12 seconds of the new agent starting, requiring a hard
reboot to recover. The root cause was not fully identified but is isolated to the BPF[^1] host-datapath program
attaching to the physical NIC[^2] (`enp1s0`, Realtek r8169) via legacy TC[^3] on Linux kernel 6.18.33. Cilium's CI test
matrix is not publicly documented, so kernel 6.18.x coverage in 1.19.4 is unverified. All clusters remain frozen at
1.18.10. No data was lost. The upgrade will be retried when Cilium 1.19.5+ or any potential fix is available on the
issue.

***

## Event Summary

**Expected outcome:** Cilium rolling upgrade completes; new 1.19 agent takes over with zero service interruption.

**Actual outcome:** Every attempt caused total packet loss within 12 seconds of the new agent attaching BPF programs to
`enp1s0`. Node fully unreachable (Talos API + Kubernetes API + iSCSI storage sessions severed). Recovery required a
physical or IPMI-level reboot each time.

**Impact:** several \~10–15 min recovery windows across 3 days. All downstream clusters lost access to OpenBao,
Pocket-Id, and the Zot registry during each window. No persistent data loss (iSCSI sessions re-established after reboot;
CNPG clusters were hibernated before last attempts).

**Duration:** 2026-05-25 to 2026-05-27 (\~5h active investigation, mostly on 26–27).

**First signal:** Node unreachable immediately after first upgrade attempt (2026-05-19).

## What Went Well

* **Rollback was always reliable.** Resetting to `quay.io/cilium/cilium:v1.18.10` + reboot consistently restored full
  cluster function. No attempt required manual BPF state cleanup.
* **CNPG databases were proactively hibernated** before May 27 attempts, preventing any risk of mid-upgrade PostgreSQL
  corruption from severed iSCSI sessions.
* **Backups completed before last session** (OpenBao + Pocket-Id snapshots).
* **Crash trigger was precisely isolated.** By attempt 7, logs confirmed the 28ms window between
  `cil_from_netdev`/`cil_to_netdev` attaching to `enp1s0` and total connectivity loss — a specific, reproducible
  signal.
* **Systematic elimination.** Eight independent hypotheses were tested and ruled out; the investigation didn't circle
  back to the same dead ends.

***

## Root-Cause Analysis

**Technique:** Ishikawa (fishbone)
**Why this technique:** 5 Whys blocked at "why does `bpf_host.c` drop all packets" — no single linear chain. Multiple interacting factors (hardware, kernel, software version, configuration, process gaps) all contributed without a single dominant cause. Ishikawa groups them by category so the structural conditions are visible even when the precise code path remains unknown.

### Analysis (Ishikawa)

Categories of factors that converged to produce the failure:

#### Hardware

* Realtek RTL8168 NIC (driver `r8169`) — not in Cilium's documented native-XDP driver list (mlx4/mlx5, ixgbe, bnxt, virtio-net, …); falls back to slower SKB-mode processing on a different code path.
* Single-node `amiya.akn` cluster — no rolling upgrade possible, no in-cluster fallback path.

#### Kernel

* Linux 6.18.33-talos — TCX (Linux 6.6+) and legacy TC both available; the 1.19 datapath chose legacy TC attach on this NIC.
* Cilium upstream CI hardware/kernel matrix is not publicly documented; r8169 + 6.18.x coverage is unverified.

#### Software (Cilium)

* Cilium 1.19.4 ships `bpf_host.c` changes that interact differently with the r8169 + 6.18 combination than 1.18.10 did. The exact code path causing total packet drop is unknown without access to Cilium's BPF source and a kernel 6.18 tracing session.
* 28ms reproducible window between TC attach to `enp1s0` and total connectivity loss — confirms the trigger surface but not the root cause inside the BPF program.

#### Configuration

* 8 hypotheses tested and ruled out: `k8sServiceHost/Port`, `bpf.enableTCX: false`, `cleanBpfState: true`, `devices: ~enp1s0`, `routingMode: native`, `bpf.hostLegacyRouting: true`, `loadBalancer.acceleration: disabled`, `enableHostLegacyRouting + bpf.lbLocalhost: false`. None restored connectivity → indicates the failure is in code, not config.

#### Process / environment

* No staging environment with matching hardware/kernel → every test was on the critical platform cluster.
* No structured log-capture protocol before each attempt → some attempts produced no usable post-mortem evidence.
* No clock-skew measurement between transcript host and cluster nodes → some timestamps unreliable.

### Root Causes

* **Cilium 1.19.4 `bpf_host.c` is incompatible with the (Linux 6.18.33 + r8169 + legacy TC) combination on this node.**
  This is a structural condition for the homelab: any future minor-version Cilium upgrade carries the same uncertainty
  until Cilium publishes a CI matrix that explicitly covers r8169 + 6.18.x or fixes the regression. The exact code
  path remains unidentified — the precise root cause inside `bpf_host.c` requires upstream-level expertise (kernel
  BPF tracing) that is out of scope for this homelab.

### Contributing Factors

* **r8169 lacks native XDP support[^4]** — Cilium's eBPF datapath prefers drivers with native XDP support (ixgbe,
  mlx4, mlx5, bnxt, virtio-net, …) and falls back to slower SKB-mode processing on others. The r8169 driver is not in
  that list. Cilium's CI hardware matrix is not publicly documented, so explicit r8169 coverage is unverified.
* **No structured timing during attempts** — wall-clock timestamps were not captured systematically, making
  cross-session analysis unreliable.
* **No staging or test environment** — all 8 attempts were made directly on the *critical* platform cluster
  (`amiya.akn`) without a prior test on equivalent hardware. This cluster hosts critical workloads (OpenBao, Pocket-Id,
  Zot) used by every other cluster in the homelab. A staging environment would have contained the failure to a
  non-production context but requires investing in duplicate hardware which is not envisaged at this time.

## Systemic Lessons

* **The platform cluster (`amiya.akn`) is a single point of failure for all clusters.** Each 10–15 min crash window
  severed OpenBao, Pocket-Id, and Zot from all downstream clusters. Eight attempts = up to 2h of cumulative platform
  unavailability. Upgrades to this cluster should have a lower risk tolerance than any application cluster.

## Agent Knowledge

* Cilium `bpf.enableTCX: false` forces legacy TC attach; `true` uses TCX (Linux 6.6+). On Linux 6.18 with r8169 (PCI 10EC:8168), legacy TC attach via `cil_from_netdev`/`cil_to_netdev` to `enp1s0` causes \~28ms-window total connectivity loss with Cilium 1.19.4. **Stay on 1.18.10 on this hardware until verified.**
* Rollback path for Cilium on `amiya.akn`: revert image tag to `quay.io/cilium/cilium:v1.18.10` + node reboot. BPF state cleanup is not required.
* `r8169` driver lacks native XDP — Cilium falls back to SKB-mode. This exercises a different code path than the documented test matrix (Mellanox mlx4/mlx5, Intel ixgbe, Broadcom bnxt, virtio-net).
* Platform-cluster upgrades (`amiya.akn`) impact every downstream cluster's access to OpenBao, Pocket-Id, and Zot. Hibernate CNPG clusters before any risky platform upgrade to prevent iSCSI-session-severance corruption.

## Footnotes

[^1]: **BPF** (Berkeley Packet Filter) — a Linux kernel technology for running sandboxed programs that filter and
    redirect network packets at the kernel level, without moving packets to userspace. Cilium uses BPF to implement all
    network policy, routing, and load balancing.
    See: [Cilium — BPF and XDP Reference](https://docs.cilium.io/en/stable/bpf/) and
    [kernel.org — BPF Documentation](https://docs.kernel.org/bpf/index.html).

[^2]: **NIC** (Network Interface Card) — the physical hardware component that connects a machine to a network.
    `enp1s0` is the Linux device name for the Realtek RTL8168-based Ethernet port on the amiya.akn node.

[^3]: **TC** (Traffic Control) — a Linux kernel subsystem for configuring packet scheduling, classification, and
    filtering on network interfaces. Cilium attaches BPF programs to TC hooks to intercept and process traffic. "Legacy
    TC" refers to the original cls\_bpf interface, as opposed to the newer TCX (TC eXpress) mechanism introduced in
    Linux 6.6+.
    See: [Linux Advanced Routing & Traffic Control HOWTO](https://lartc.org/howto/) and
    [Cilium — TCX Guide](https://docs.cilium.io/en/stable/network/tcx/).

[^4]: **XDP driver support** — XDP requires the network driver to implement the RX hook. The community-maintained
    [XDP Driver Support reference][xdp-drivers] enumerates Mellanox mlx4/mlx5, Netronome nfp, virtio-net,
    Cavium/Qlogic qede, Cavium thunder/nicvf, Broadcom bnxt, and Intel ixgbe — r8169 (Realtek RTL8168) is not listed.
    Drivers without native XDP fall back to generic (`SKB`/`xdpgeneric`) mode, which is slower and exercises a
    different code path.

[xdp-drivers]: https://prototype-kernel.readthedocs.io/en/latest/networking/XDP/implementation/drivers.html
